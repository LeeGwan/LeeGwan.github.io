## [0x01] What Intel Pin is, and why I built this with it

**[-] Intel Pin = a Dynamic Binary Instrumentation (DBI) tool**

When you analyze with a debugger you set breakpoints, stop, and inspect memory. But something like "give me every memory address this function touched" is far too tedious to do one breakpoint at a time. Pin automates exactly this. Without modifying the target EXE, it creates points where execution enters your own code in between the running instructions, reads or records the register/memory values at that moment, and then returns to the original flow. No source, no recompilation required.

**[-] Why I built this tool**

When you trace real virtualized code by hand, alongside the meaningful `vm_handler`s there's a pile of fake handlers, obfuscation, and anti-debugging mixed in. That noise slows reversing down sharply and wears you out fast. So I thought: "Why not just run it once and dump, as a log, roughly how this VM operates ‚Äî which handler touches what memory ‚Äî so it becomes a map of where to actually look when I analyze for real later." This post's tool started from there.

**[-] Pin works through callbacks**

Pin is designed around callbacks. When you register an instrumentation function like `INS_AddInstrumentFunction`, Pin shows you each instruction once when it JIT-compiles the code. The point to watch is that you don't analyze right there ‚Äî instead you attach, with `INS_InsertCall`, a callback that runs *every time it executes*. To put it together: the registration-side functions (`IMG_Callbackfunc`, `INS_Callbackfunc`, `AnalyzeInstruction`) sweep the instructions and only decide "where to insert which callback," while the callbacks attached there (`OnHandlerEnter`, `OnMemRead`, `OnMemWriteAfter`) fire on every execution to print the actual values.

**[-] IPOINT and IARG**

The callback you attach specifies where, relative to the instruction, it gets inserted (`IPOINT_BEFORE` just before execution / `IPOINT_AFTER` just after / `IPOINT_TAKEN_BRANCH` the path where the branch is taken), and receives the values it needs via `IARG_*` argument specifiers. For example, `IARG_REG_VALUE, REG_EAX` means "pass me the value of EAX," and `IARG_MEMORYREAD_EA` means "pass me the address of this read." The argument list always ends with `IARG_END`.

## [0x02] The VMCONTEXT struct (PIN_CLASS.h)

First, the header. This simply ports the `VM_Context` we dumped in Part 2 (base `0x41E600`) into a struct.

```cpp
#pragma pack(push, 1)
typedef struct _VMCONTEXT
{
    uint32_t VM_EBP;                // +0x00: virtual EBP
    uint32_t VM_EDI;                // +0x04: virtual EDI
    uint32_t VM_EBX;                // +0x08: virtual EBX
    uint32_t VM_EAX;                // +0x0C: virtual EAX
    uint32_t VM_ESI;                // +0x10: virtual ESI
    uint32_t VM_EDX;                // +0x14: virtual EDX
    uint32_t VM_ECX;                // +0x18: virtual ECX
    uint32_t VM_EFLAGS;             // +0x1C: virtual EFLAGS
    uint32_t unk_20;                // +0x20: unknown
    uint32_t unk_24;                // +0x24: unknown
    uint32_t unk_28;                // +0x28: unknown
    uint32_t unk_2C;                // +0x2C: unknown
    uint32_t init_flag;             // +0x30: Control Code lock cmpxchg target
    uint32_t unk_34;                // +0x34: unknown
    ADDRINT* handler_table[0x96];   // +0x38: array of 0x96 handler addresses
} VMCONTEXT, *LPVMCONTEXT;
#pragma pack(pop)
```

Line by line:

- **`VM_EBP` ~ `VM_EFLAGS` (+0x00 ~ +0x1C)** ‚Äî the 8 virtual register slots confirmed in Part 2. The Real CPU values backed up by `pushad`/`pushfd` get copied here.
- **`unk_20` ~ `unk_34`** ‚Äî slots whose purpose is less clear. They're named just to fill the space so the struct offsets stay unbroken.
- **`init_flag` (+0x30)** ‚Äî the slot the Control Code touches via `lock cmpxchg [edi+30]`. You can see it being touched directly at the very end of the trace, at `[0x47]`.
- **`handler_table[0x96]` (+0x38)** ‚Äî the array of Opcode handler addresses. The Main Handler indexes this table with `jmp [edi+eax*4]` and branches. The count `0x96` is the handler-table size confirmed in Part 2.

Below is the actual `0x41E600` dump. Picture the struct above overlaid directly on this memory.

![VM_Context memory dump](/assets/img/Code Virtualization/Code Virtualization-1/vm_context_after.png)

*0x41E600 ‚Äî the front holds the register slots (+0x00~+0x1C); from +0x38 onward is the handler-address table*

The first row of `0x41E600`, `E4 FE 19 00`, gives `VM_EBP = 0019FEE4`; `16 02 00 00` at `+0x1C` gives `VM_EFLAGS = 0x216`; and `0E 0D 42 00 ‚Ä¶` from `+0x38` lands as `handler_table[0] = 0x00420D0E`. You need this base address (`g_vmContextAddr`) before classifications like "this address is the `VM_EAX` slot" become possible later.

## [0x03] Entry point and callback registration

`main` does little more than create one class object, initialize it, and start.

```cpp
int main(int argc, char* argv[])
{
    PIN_CLASS g_bPIN_CLASS;
    if (!g_bPIN_CLASS.pinClassInit(argc, argv))   // exit on init failure
        return -1;
    g_bPIN_CLASS.pinClassStart();                 // start instrumenting (does not return)
    return 0;
}
```

The constructor calls `PIN_InitSymbols()`. This prepares for using symbol information (function names, etc.); by Pin convention it must be called before `PIN_Init`.

```cpp
PIN_CLASS::PIN_CLASS() { PIN_InitSymbols(); }
```

Initialization centers on calling `PIN_Init`.

```cpp
bool PIN_CLASS::pinClassInit(int argc, char* argv[])
{
    if (PIN_Init(argc, argv) != FALSE) { std::cout << "[PIN] Init failed\n"; return false; }
    std::cout << "[PIN] Init success\n";
    return true;
}
```

- **`PIN_Init(argc, argv)`** ‚Äî initializes the Pin system and parses the command line. The easy thing to get wrong here is **what the return value means**. `PIN_Init` returns TRUE *if an error occurred*. So when it's `!= FALSE` (an error occurred), we treat it as failure and return `false`. It's the opposite of intuition, so watch out.

Now we register callbacks and start the program.

```cpp
void PIN_CLASS::pinClassStart()
{
    IMG_AddInstrumentFunction(IMG_Callbackfunc, this);   // image-load callback
    INS_AddInstrumentFunction(INS_Callbackfunc, this);   // instruction instrumentation callback
    PIN_AddFiniFunction(Fini_Callbackfunc, this);        // fini callback
    PIN_StartProgram();                                  // never returns
}
```

- **`IMG_AddInstrumentFunction(fun, val)`** ‚Äî registers `fun` to be called every time an image (EXE/DLL) is loaded. The second argument `val` is passed straight back when the callback is invoked. We pass `this` because the callback is a `static` function and can't access members directly; receiving `this` lets the callback cast back to the instance and call member functions.
- **`INS_AddInstrumentFunction`** ‚Äî registers the per-instruction instrumentation callback. (also passing `this`)
- **`PIN_AddFiniFunction`** ‚Äî a callback to be invoked just before the program exits. We print the handler summary here.
- **`PIN_StartProgram()`** ‚Äî starts the target running. This function **does not return** (it even unwinds the stack). So the trailing `return 0;` is effectively unreachable.

## [0x04] The three callbacks

```cpp
VOID PIN_CLASS::IMG_Callbackfunc(IMG img, VOID* v)
{
    if (!IMG_IsMainExecutable(img)) return;            // main EXE only
    static_cast<PIN_CLASS*>(v)->ProcessImage(img);     // v = the this we passed earlier
}
```

- **`IMG_IsMainExecutable(img)`** ‚Äî checks whether the image being loaded is the main executable. The VM section (`.v-lizer`) lives inside the target EXE, so we filter out the system DLLs that get pulled in.
- **`static_cast<PIN_CLASS*>(v)`** ‚Äî restores the `this` passed in `pinClassStart` back to an instance, to call the member function `ProcessImage`.

```cpp
VOID PIN_CLASS::INS_Callbackfunc(INS ins, VOID* v)
{
    ADDRINT ip = INS_Address(ins);
    if (!g_vmStart || !g_vmEnd || ip < g_vmStart || ip > g_vmEnd) return;  // ignore if outside VM range
    static_cast<PIN_CLASS*>(v)->AnalyzeInstruction(ins);
}
```

- **`INS_Address(ins)`** ‚Äî gets this instruction's address (IP).
- **Range gate** ‚Äî if `g_vmStart`/`g_vmEnd` aren't set yet (`!g_vmStart`), or the IP is outside the VM section, we bail immediately. The point is to instrument **only the instructions inside the VM region**, not the whole target. Otherwise a callback gets attached across the entire program, which is slow and floods the log.

```cpp
VOID PIN_CLASS::Fini_Callbackfunc(INT32 code, VOID* v)
{
    DumpHandlerSummary();
    std::cout << "\n[PIN] target exited (code " << code << ")\n";
}
```

On exit it prints the handler summary table and the exit code.

## [0x05] Detecting the VM region (ProcessImage)

```cpp
void PIN_CLASS::ProcessImage(const IMG& img)
{
    for (SEC sec = IMG_SecHead(img); SEC_Valid(sec); sec = SEC_Next(sec))
        if (SEC_Name(sec) == kVmSection)              // ".v-lizer"
        {
            g_vmStart = SEC_Address(sec);
            g_vmEnd   = g_vmStart + SEC_Size(sec);
            std::cout << "[PIN] " << kVmSection << " @ " << StringFromAddrint(g_vmStart) << "\n";
            break;
        }
}
```

- **`IMG_SecHead` / `SEC_Valid` / `SEC_Next`** ‚Äî the standard pattern for iterating an image's sections from first to last. When `SEC_Valid` is false, you've reached the end.
- **`SEC_Name(sec) == kVmSection`** ‚Äî compares the section name against the VM section we're looking for (`.v-lizer` in this sample). Oreans-family virtualization section names differ per sample, so for a different target you only need to change this constant.
- **`SEC_Address` / `SEC_Size`** ‚Äî takes that section's start address and size to fix the `[g_vmStart, g_vmEnd]` range. This range is used as the gate in `INS_Callbackfunc` above.

## [0x06] State control ‚Äî PUSHAD/POPAD

The VM enters via the Control Code's `pushad`/`pushfd` and leaves via `popad`/`popfd` (see Parts 1 and 2). We use that boundary directly as a switch.

```cpp
VOID PIN_CLASS::TurnOnVmSwitch()
{
    g_vmActive = true;
    // safely read the vm_context pointer stored at (VM section start + 0x8a3)
    PIN_SafeCopy(&g_vmContextAddr,
                 (VOID*)(g_vmStart + kVmContextPtrOffset),
                 sizeof(g_vmContextAddr));
    g_vmContext = reinterpret_cast<VMCONTEXT*>(g_vmContextAddr);
}
VOID PIN_CLASS::TurnOffVmSwitch() { g_vmActive = false; }
BOOL PIN_CLASS::CheckIfVmActive()  { return g_vmActive ? TRUE : FALSE; }
```

- **`g_vmActive = true`** ‚Äî the tracing-on flag. Only while this is on does `AnalyzeInstruction` insert analysis routines.
- **`PIN_SafeCopy(dst, src, size)`** ‚Äî safely copies memory from `src` to `dst`. Unlike a plain `memcpy`, even an inaccessible address won't crash; it just returns the number of bytes copied. Always use this when reading runtime memory.
  - Here it reads 4 bytes from `g_vmStart + 0x8a3` into `g_vmContextAddr`. **0x8a3** is, in this sample, the offset from the VM section start to where vm_context (`0x41E600`) is stored. We must read this value before region classification becomes possible. (For a different sample this offset will differ too.)
- **`reinterpret_cast<VMCONTEXT*>`** ‚Äî takes the read address as a struct pointer, so its address value can be accessed when needed.

`POPAD` (exit) is turned off conditionally.

```cpp
else if (op == XED_ICLASS_POPAD)
{
    INS_InsertIfCall(ins,   IPOINT_BEFORE, (AFUNPTR)CheckIfVmActive, IARG_END);
    INS_InsertThenCall(ins, IPOINT_BEFORE, (AFUNPTR)TurnOffVmSwitch, IARG_END);
}
```

- **`INS_InsertIfCall` + `INS_InsertThenCall`** ‚Äî these work as a pair. Only when the "If" routine (`CheckIfVmActive`) returns a non-zero value does the immediately following "Then" routine (`TurnOffVmSwitch`) run. In other words, **only a `popad` while the VM is active** is treated as an exit. Put simply: if `pushad` is the start point of virtualization (or VM processing), then `popad` can be seen as the point where, virtualization processing having finished, the saved Real CPU register values are restored.

## [0x07] Instruction instrumentation (AnalyzeInstruction)

While the VM is active, this is the core that decides which analysis routine to insert for each instruction inside the VM range. First, the enter/exit switches.

```cpp
void PIN_CLASS::AnalyzeInstruction(const INS& ins)
{
    OPCODE op = INS_Opcode(ins);

    if (op == XED_ICLASS_PUSHAD)
        INS_InsertCall(ins, IPOINT_BEFORE, (AFUNPTR)TurnOnVmSwitch, IARG_END);
    else if (op == XED_ICLASS_POPAD)
    {
        INS_InsertIfCall(ins,   IPOINT_BEFORE, (AFUNPTR)CheckIfVmActive, IARG_END);
        INS_InsertThenCall(ins, IPOINT_BEFORE, (AFUNPTR)TurnOffVmSwitch, IARG_END);
    }
    ...
```

- **`INS_Opcode(ins)`** ‚Äî gets the kind (opcode) of the instruction.
- **`PUSHAD` ‚Üí `TurnOnVmSwitch` at `IPOINT_BEFORE`** ‚Äî turns the switch on *just before* `pushad` executes.
- **`POPAD` ‚Üí `InsertIfCall(CheckIfVmActive)` + `InsertThenCall(TurnOffVmSwitch)`** ‚Äî `popad` appears when the VM restores registers and leaves.

### [-] Handler entry ‚Äî placing a callback on the dispatcher jump

Part 2's Main Handler indexes the handler table and branches with `jmp [edi+eax*4]`. We pinpoint exactly that pattern.

```cpp
if (INS_IsIndirectControlFlow(ins) && !INS_IsRet(ins) &&
    INS_IsValidForIpointTakenBranch(ins) &&
    INS_MemoryBaseReg(ins)  == REG_EDI &&
    INS_MemoryIndexReg(ins) == REG_EAX &&
    INS_MemoryScale(ins)    == 4)
{
    INS_InsertCall(ins, IPOINT_TAKEN_BRANCH, (AFUNPTR)OnHandlerEnter,
        IARG_REG_VALUE, REG_EAX,        // decoded opcode (table index)
        IARG_BRANCH_TARGET_ADDR,        // the actual handler address being jumped to
        IARG_END);
}
```

Each condition is a filter to pin down `jmp [edi+eax*4]` exactly.

- **`INS_IsIndirectControlFlow`** ‚Äî is it indirect control flow? Whether `JMP`, `CALL`, or `ret`, it's true for any control-flow instruction whose **branch target comes from a register or memory**. `jmp [edi+eax*4]`, where the target isn't an immediate / IP-relative offset but is read from memory, falls here.
- **`!INS_IsRet`** ‚Äî `ret` takes its target from the stack (memory), so it also matches the condition above. Since what we want is the dispatcher jump, not a function return, we exclude `ret`.
- **`INS_IsValidForIpointTakenBranch`** ‚Äî a function that asks in advance whether we can insert an analysis routine at the `IPOINT_TAKEN_BRANCH` location for this instruction. `IPOINT_TAKEN_BRANCH` is the point that fires a call only on "the path where the branch is *actually taken*" ‚Äî i.e., the exact moment control transfers to the target ‚Äî and that's only meaningful for branch-family instructions. Attaching `IPOINT_TAKEN_BRANCH` to a non-branch instruction makes Pin throw an error. So we check this is true before calling `INS_InsertCall`, ensuring we only attach a callback to the dispatcher jump we're after.
- **`MemoryBaseReg == REG_EDI` / `MemoryIndexReg == REG_EAX` / `MemoryScale == 4`** ‚Äî checks that the memory operand is exactly the `[EDI + EAX*4]` form. base=EDI, index=EAX, scale=4 ‚Äî that is the handler-table indexing pattern. These three conditions distinguish it from other indirect jumps.

The callback is placed at `IPOINT_TAKEN_BRANCH` (the path where the branch is actually taken). That way we catch the very moment control crosses into the handler. The arguments:

- **`IARG_REG_VALUE, REG_EAX`** ‚Äî the EAX value at that moment. The opcode (table index) the Main Handler finished decoding sits in EAX.
- **`IARG_BRANCH_TARGET_ADDR`** ‚Äî the target address this branch jumps to, i.e., the actual handler address. Pin resolves `[edi+eax*4]` for us so we don't compute it ourselves.

### [-] LEA / read / write

```cpp
if (INS_IsLea(ins)) {
    REG  baseReg    = INS_MemoryBaseReg(ins);
    BOOL isStackLea = (baseReg == REG_ESP || baseReg == REG_EBP);
    INS_InsertPredicatedCall(ins, IPOINT_BEFORE, (AFUNPTR)OnLea,
        IARG_EXPLICIT_MEMORY_EA,                 // the address lea computed
        IARG_REG_VALUE, REG_ESP, IARG_BOOL, isStackLea, IARG_END);
}
```

- **`INS_IsLea`** ‚Äî is it a `lea`? `lea` only computes an address; it doesn't read memory. So it has to be caught separately from the memory read/write callbacks. (I initially did this with `INS_IsMemoryRead` and ran into problems, so I switched to this.)
- **`IARG_EXPLICIT_MEMORY_EA`** ‚Äî receives the effective address `lea` computed. Pin documents this argument specifically for instrumenting `lea`, so it cleanly grabs the computed address without any (non-existent) "memory access."
- **`isStackLea`** ‚Äî if base is ESP/EBP we treat it as a stack-address computation and pass a flag (for log labeling).
- **`INS_InsertPredicatedCall`** ‚Äî almost the same as `INS_InsertCall`, but for instructions with a predicate (CMOVcc, REP, etc.) it skips the analysis routine when the condition is false. Put simply, it makes the instrumentation fire only when the memory access actually happened (`lea`, `read`, `write`).

```cpp
if (INS_IsMemoryRead(ins)) {
    BOOL isStackRead = INS_IsStackRead(ins);
    INS_InsertPredicatedCall(ins, IPOINT_BEFORE, (AFUNPTR)OnMemRead,
        IARG_MEMORYREAD_EA, IARG_MEMORYREAD_SIZE,   // read address + size
        IARG_REG_VALUE, REG_ESP, IARG_BOOL, isStackRead, IARG_END);
}
```

- **`INS_IsMemoryRead`** ‚Äî is it an instruction that reads memory?
- **`IARG_MEMORYREAD_EA` / `IARG_MEMORYREAD_SIZE`** ‚Äî valid only at `IPOINT_BEFORE`. It is the address at which the memory read occurred.

```cpp
if (INS_IsMemoryWrite(ins)) {
    BOOL isStackWrite = INS_IsStackWrite(ins);
    INS_InsertPredicatedCall(ins, IPOINT_BEFORE, (AFUNPTR)OnMemWriteBefore,
        IARG_MEMORYWRITE_EA, IARG_MEMORYWRITE_SIZE,  // write address + size (value not yet available)
        IARG_REG_VALUE, REG_ESP, IARG_END);

    if (INS_IsValidForIpointAfter(ins))
        INS_InsertPredicatedCall(ins, IPOINT_AFTER,
            (AFUNPTR)OnMemWriteAfter, IARG_BOOL, isStackWrite, IARG_END);
}
```

Why the write is split into BEFORE/AFTER is the key point.

- **`IARG_MEMORYWRITE_EA` is valid only at `IPOINT_BEFORE`**. But at that point the value hasn't been written to memory yet. So at BEFORE (`OnMemWriteBefore`) we only stash **the address, size, and ESP**.
- The actually-stored value is confirmed at AFTER (`OnMemWriteAfter`) by re-reading that address. We attach the callback only when `INS_IsValidForIpointAfter` says this instruction supports IPOINT_AFTER (branches, `call`, etc. have no fall-through, so it can't be used).

## [0x08] Analysis routines and region classification

### [-] Handler entry log (OnHandlerEnter)

```cpp
VOID PIN_CLASS::OnHandlerEnter(ADDRINT opcode, ADDRINT target)
{
    g_curHandler = target;
    g_curOpcode  = opcode;
    g_inDispatch = false;              // handler body begins ‚Üí filtering off (the logs we want start here)

    HandlerInfo& h = g_handlerStats[target];
    h.callCount++;
    if (h.sampleOpcode.empty()) h.sampleOpcode = StringFromAddrint(opcode);

    LOG("[" + StringFromAddrint(g_step++) + "] op(DECODING)=" +
        StringFromAddrint(opcode) + " handler=" + StringFromAddrint(target) + "\n");
}
```

- **`opcode`/`target`** ‚Äî the `IARG_REG_VALUE REG_EAX` and `IARG_BRANCH_TARGET_ADDR` passed above come in here directly.
- **Updating `g_curHandler`/`g_curOpcode`** ‚Äî the current state values that mark "which handler" the reads/writes from now on belong to.
- **`g_inDispatch = false`** ‚Äî the handler body has begun, so we turn filtering off. What gets printed from here is the log we want to see. Conversely, when the garbage values the decoder produces are detected, `g_inDispatch = true` turns filtering on, and that span is not stored in the log (explained below in [0x09]).
- **`g_handlerStats[target]`** ‚Äî accumulates statistics keyed by handler address. `callCount++` for call count, and on first appearance records the opcode in `sampleOpcode`.
- **`LOG(...)`** ‚Äî the trace's `[0x..] op(DECODING)=.. handler=..` header line comes from here. `g_step++` numbers the steps.

### [-] Region classification (RegionName)

It classifies whether the address being read/written is a vm_context slot, the stack, or VCODE, and attaches a label. This is what makes the log readable.

```cpp
static std::string RegionName(ADDRINT x, ADDRINT esp, BOOL isStack)
{
    if (g_vmContextAddr && x >= g_vmContextAddr &&
        x < g_vmContextAddr + sizeof(VMCONTEXT))     // inside the vm_context range?
    {
        UINT32 off = (UINT32)(x - g_vmContextAddr);  // offset from the base
        switch (off) {
        case 0x00: return "VM_EBP";  case 0x04: return "VM_EDI";
        case 0x08: return "VM_EBX";  case 0x0C: return "VM_EAX";
        case 0x10: return "VM_ESI";  case 0x14: return "VM_EDX";
        case 0x18: return "VM_ECX";  case 0x1C: return "VM_EFLAGS";
        case 0x30: return "init_flag";
        default:
            if (off >= 0x38) return "handler_table";
            return "VMCTX+" + StringFromAddrint(off);
        }
    }
    if (isStack) return "STACK";
    if (g_vmStart && x >= g_vmStart && x < g_vmEnd) return "Read OPCODE";
    return "";
}
```

- **vm_context range check** ‚Äî if address `x` is inside `[g_vmContextAddr, +sizeof(VMCONTEXT))`, it's a vm_context slot.
- **Mapping a slot name via `off`** ‚Äî using the offset from the base, it returns `VM_EAX`, `VM_EFLAGS`, `init_flag`, etc. `0x38` or above is `handler_table`; other unnamed slots are printed as `VMCTX+offset`.
- **`STACK`** ‚Äî if it isn't vm_context and is a stack access, it's `STACK`.
- **`Read OPCODE`** ‚Äî a read inside the VM section (`[g_vmStart, g_vmEnd)`) is treated as a bytecode fetch and labeled `Read OPCODE`.
- **Otherwise** ‚Äî returns an empty string, which the caller fills in as `MEM` (heap/target data).

There's a reason we track the stack (especially ESP-relative access) separately. After a `vm_handler` processes an operation, it handles the result in one of two ways. One is writing the value into a specific vm_context slot, finishing the operation in place; the other is **pushing the result or an intermediate value onto the stack (specifically onto ESP)** to hand it to the next handler. In the latter case, the next handler pops the value one handler placed on the ESP side and processes it ‚Äî so the handlers exchange data through the stack. That's why labeling these stack-slot writes/reads as `STACK` makes the inter-handler data flow ‚Äî "which handler pushed what, and which handler picks it up" ‚Äî visible in the log.

Two functions wrap this classification.

```cpp
static std::string Loc(ADDRINT addr, ADDRINT esp, BOOL isStack) {
    std::string r = RegionName(addr, esp, isStack);
    if (r.empty()) r = "MEM";
    return "[" + r + "]" + StringFromAddrint(addr);   // e.g. [VM_EAX]0x0041e60c
}
static std::string ValTag(ADDRINT val, ADDRINT esp, BOOL isStack) {
    std::string r = RegionName(val, esp, isStack);
    return r.empty() ? "" : " (->" + r + ")";          // e.g. (->STACK)
}
```

- **`Loc`** ‚Äî labels an address in the form `[region]0xADDR`. On classification failure, `MEM`.
- **`ValTag`** ‚Äî if the *value* itself points to another region, it attaches a `(->STACK)` or `(->VM_EAX)` tag. This makes it easy to follow pointer chains where a slot address flows through the stack.

### [-] Read log

`lea` doesn't read memory ‚Äî it only computes an address ‚Äî so it looks almost like a read, but with no *value*, printing only the *computed address*.

```cpp
VOID PIN_CLASS::OnLea(ADDRINT ea, ADDRINT esp, BOOL isStack)
{
    if (!g_curHandler) return;                    // ignore before any handler entry
    if (g_inDispatch) return;                     // if decode noise, skip printing only

    LOG("      LEA   -> " + Loc(ea, esp, isStack) + "\n");
}
```

- **`ea`** ‚Äî the effective address `lea` computed, passed from the instrumentation side via `IARG_EXPLICIT_MEMORY_EA`. Since it doesn't touch memory there's no value to read, so there's no `val:` and no `ValTag`.
- **`Loc(ea, ...)`** ‚Äî classifies whether that address is a vm_context slot or the stack and prints it like `LEA -> [VM_EAX]0x0041e60c`.

```cpp
VOID PIN_CLASS::OnMemRead(ADDRINT addr, UINT32 size, ADDRINT esp, BOOL isStack)
{
    if (!g_curHandler) return;                    // ignore before any handler entry
    g_handlerStats[g_curHandler].readCount++;     // accumulate stats
    if (g_inDispatch) return;                     // if decode noise, skip printing only

    ADDRINT val = 0;
    PIN_SafeCopy(&val, (VOID*)addr, (size < sizeof(ADDRINT)) ? size : sizeof(ADDRINT));
    LOG("      READ  " + Loc(addr, esp, isStack) +
        " val:" + StringFromAddrint(val) + ValTag(val, esp, isStack) + "\n");
}
```

- **`if (!g_curHandler) return`** ‚Äî if we haven't entered a handler yet (the dispatcher prologue, etc.), ignore it. (`OnLea` keeps the same guard.)
- **`readCount++`** ‚Äî accumulates the read count for the current handler. (`lea` is not a memory access, so it doesn't raise the read count.)
- **Reading the value with `PIN_SafeCopy`** ‚Äî safely reads the value at `addr`. The size is **clamped** with `(size < 4) ? size : 4`. This prevents reading a full 4 bytes on a 1/2-byte access and dragging in adjacent memory.
- **`LOG`** ‚Äî prints one line `READ [region]addr val:value (->region)`, using `Loc` for the address and `ValTag` for the region the value points to.

### [-] Write log

The write stashes only the coordinates at BEFORE:

```cpp
VOID PIN_CLASS::OnMemWriteBefore(ADDRINT addr, UINT32 size, ADDRINT esp)
{
    g_wAddr = addr; g_wSize = size; g_wEsp = esp;   // value not written yet ‚Üí keep only the coordinates
}
```

and at AFTER reads the actually-stored value and prints it (including the noise check, next section).

At first I logged every memory access, and the garbage values produced by the Main Handler's fetch/decode routine plastered the log. So we detect the **signature values** the decode routine writes to the stack, and skip printing until the next handler entry.

```cpp
static const ADDRINT kDispatchMagic1 = 0x4b82;
static const ADDRINT kDispatchMagic2 = 0x25c5;

VOID PIN_CLASS::OnMemWriteAfter(BOOL isStack)
{
    if (!g_curHandler) return;
    g_handlerStats[g_curHandler].writeCount++;     // accumulate stats

    ADDRINT val = 0;
    PIN_SafeCopy(&val, (VOID*)g_wAddr, (g_wSize < sizeof(ADDRINT)) ? g_wSize : sizeof(ADDRINT));

    // if the written value is a decoder signature, treat up to the next handler entry as noise
    if (val == kDispatchMagic1 || val == kDispatchMagic2)
        g_inDispatch = true;
    if (g_inDispatch) return;                      // skip printing only

    LOG("      WRITE " + Loc(g_wAddr, g_wEsp, isStack) +
        " <- " + StringFromAddrint(val) + ValTag(val, g_wEsp, isStack) + "\n");
}
```

- **Reading the value with the coordinates stashed at BEFORE (`g_wAddr`, etc.)** ‚Äî now that the write is done, re-reading that address yields the *actually-stored value*. That's why the value printed in the log is the true result of the store.
- **Signature detection** ‚Äî if the written value is `0x4b82`/`0x25c5`, we consider the decoder to have started and turn filtering on with `g_inDispatch = true`. The garbage values in this span are not stored in the log.
- **Skip printing while `g_inDispatch`** ‚Äî this flag is cleared back to `false` at the next `OnHandlerEnter`. So decode noise gets buried, and only the meaningful accesses in the handler body remain in the trace.

 **Note:** the counter increments (`readCount++`/`writeCount++`) come **before** the `g_inDispatch` check. So the reads/writes in the exit summary are raw counts that include the noise, and may be larger than the number of lines actually printed in the trace.

Finally, the summary output.

```cpp
VOID PIN_CLASS::DumpHandlerSummary()
{
    LOG("\n================= HANDLER SUMMARY =================\n");

```

## [0x0B] Conclusion

What used to be 72 steps followed by hand is now, with a single run, reduced to printing "which handler, how many times, what." On top of Pin's instruction instrumentation, layering just three things ‚Äî the dispatcher callback (`jmp [edi+eax*4]` + `IPOINT_TAKEN_BRANCH`), region classification (`VM_Context`/`STACK`/`Read OPCODE`/`MEM`), and noise filtering (signatures `0x4b82`/`0x25c5`) ‚Äî was enough to pull a trace buried in garbage into a readable form.

Of course, this tool only produces a clean trace; it doesn't tell you the meaning of each handler. From there you have to follow along and reconstruct it yourself, like in Part 2.
