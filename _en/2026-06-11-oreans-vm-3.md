When analyzing software using standard debuggers, you typically set breakpoints, pause execution, and inspect memory manually. However, tasks like dumping every unique memory address touched by a specific routine are highly tedious using breakpoints alone. Intel Pin automates this. Without modifying the target binary or requiring its source code/recompilation, Pin sits between the binary and the CPU, injecting monitoring hooks to inspect or log register states and memory values at runtime before resuming normal execution.**[-] Motivation for Building This Tool**



When reversing virtualized code manually, you encounter massive noise introduced by fake handlers, heavy obfuscation, and convoluted dispatch loops alongside the actual meaningful `vm_handler` routines. This noise drastically bogs down analysis velocity. The motivation behind this tool was straightforward: if we can execute the binary once and map out the core VM layout—identifying which handlers touch which memory regions—we generate an invaluable map to guide our deep manual reversing later.**[-] Callback-Driven Architecture of Pin**



Intel Pin relies on a callback-driven paradigm. By registering an instrumentation opportunity using functions like `INS_AddInstrumentFunction`, Pin presents each instruction to your tool during its JIT (Just-In-Time) compilation phase. Crucially, you do not analyze the context directly inside this compilation function; instead, you use `INS_InsertCall` to inject the *actual analysis callbacks that run during execution*. In short, the setup functions (`IMG_Callbackfunc`, `INS_Callbackfunc`, `AnalyzeInstruction`) inspect the instruction stream to decide *what to hook*, and the injected analysis routines (`OnHandlerEnter`, `OnMemRead`, `OnMemWriteAfter`) log the real values during execution.**[-] Instrumentation Points (IPOINT) and Arguments (IARG)**



When injecting an analysis routine, you specify exactly when it executes relative to the instruction: `IPOINT_BEFORE` (immediately before execution), `IPOINT_AFTER` (immediately after execution, if fall-through is valid), or `IPOINT_TAKEN_BRANCH` (only when a branch is taken). The runtime data is requested via `IARG_*` constants. For example, `IARG_REG_VALUE, REG_EAX` commands Pin to pass the current value of EAX, and `IARG_MEMORYREAD_EA` requests the effective address of a memory read. The argument list must always terminate with `IARG_END`.## [0x02] VMCONTEXT Structure (PIN_CLASS.h)

    uint32_t VM_ECX;                // +0x18: Virtual ECX

    uint32_t VM_EFLAGS;             // +0x1C: Virtual EFLAGS

    uint32_t unk_20;                // +0x20: Unknown

    uint32_t unk_24;                // +0x24: Unknown

    uint32_t unk_28;                // +0x28: Unknown

    uint32_t unk_2C;                // +0x2C: Unknown

    uint32_t init_flag;             // +0x30: Target of Control Code lock cmpxchg

Breakdown of the fields:



VM_EBP to VM_EFLAGS (+0x00 to +0x1C) — The 8 virtual register slots identified in Part 2. The Real CPU states backed up by pushad/pushfd are transferred here.

unk_20 to unk_34 — Reserved fields whose precise purposes remain unconfirmed. They are retained to preserve exact structure alignment and matching offsets.

init_flag (+0x30) — The synchronization target modified by the Control Code via lock cmpxchg [edi+30]. This location is prominently modified in step [0x47] of our trace.

handler_table[0x96] (+0x38) — The array of Opcode handler entry points. The Main Handler indexes into this table using jmp [edi+eax*4] to dispatch execution. The array bound 0x96 matches the exact size of the handler table determined in Part 2.

[0x04] The Three Primary Callbacks

C++



VOID PIN_CLASS::IMG_Callbackfunc(IMG img, VOID* v){

    if (!IMG_IsMainExecutable(img)) return;            // Isolate the main binary

    static_cast<PIN_CLASS*>(v)->ProcessImage(img);     // Cast back to our instance

}

IMG_IsMainExecutable(img) — Restricts parsing strictly to the primary target binary. Since our virtualized code section (.v-lizer) resides exclusively inside the main executable, this filter discards incoming system libraries or runtime DLL noise.

static_cast<PIN_CLASS*>(v) — Restores our opaque this pointer, safely transferring the execution context back into the object-oriented ProcessImage worker.

C++



VOID PIN_CLASS::INS_Callbackfunc(INS ins, VOID* v){

    ADDRINT ip = INS_Address(ins);

    if (!g_vmStart || !g_vmEnd || ip < g_vmStart || ip > g_vmEnd) return;  // Out of VM scope

    static_cast<PIN_CLASS*>(v)->AnalyzeInstruction(ins);

}

INS_Address(ins) — Retrieves the raw instruction pointer (IP) for the instruction currently undergoing compilation.

The Range Gate — If our boundary variables aren't established yet (!g_vmStart) or if the current IP falls entirely outside the bounded .v-lizer boundaries, we return immediately. Only instructions executing inside the virtualized zone undergo instrumentation. Without this gate, Pin would instrument every routine inside standard libraries (like ntdll.dll), decimating performance and flooding log files.

C++

[0x05] VM Region Target Tracking (ProcessImage)

C++



void PIN_CLASS::ProcessImage(const IMG& img){

    for (SEC sec = IMG_SecHead(img); SEC_Valid(sec); sec = SEC_Next(sec))

        if (SEC_Name(sec) == kVmSection)              // Match ".v-lizer"

        {

            g_vmStart = SEC_Address(sec);

            g_vmEnd   = g_vmStart + SEC_Size(sec);

            std::cout << "[PIN] " << kVmSection << " @ " << StringFromAddrint(g_vmStart) << "\n";

            break;

        }

}

IMG_SecHead / SEC_Valid / SEC_Next — Standard iteration loop to parse sections within an image container. It breaks as soon as SEC_Valid evaluates to false.

SEC_Name(sec) == kVmSection — Compares the section name token against our target virtualized section designator (.v-lizer). Because protection engines vary section tokens between compilations, this is the configuration variable you alter when targeting different binaries.

SEC_Address / SEC_Size — Captures the virtual base and physical span of the section, storing the results in [g_vmStart, g_vmEnd]. This explicitly establishes the range gate checked inside INS_Callbackfunc.

[0x06] State Control — PUSHAD/POPAD Boundaries

The virtualized matrix begins execution through a Control Code pushad/pushfd sequence and leaves via a matching popad/popfd execution sequence. We track these locations to toggle our dynamic tracer.



C++



VOID PIN_CLASS::TurnOnVmSwitch(){

    g_vmActive = true;

    // Extract the live pointer to the vm_context stored at VM Section Base + 0x8a3

    PIN_SafeCopy(&g_vmContextAddr,

                 (VOID*)(g_vmStart + kVmContextPtrOffset),

                 sizeof(g_vmContextAddr));

    g_vmContext = reinterpret_cast<VMCONTEXT*>(g_vmContextAddr);

}VOID PIN_CLASS::TurnOffVmSwitch() { g_vmActive = false; }BOOL PIN_CLASS::CheckIfVmActive()  { return g_vmActive ? TRUE : FALSE; }

g_vmActive = true — Toggles active tracing. While true, AnalyzeInstruction injects our heavy monitoring analysis callbacks.

PIN_SafeCopy(dst, src, size) — Safely copies memory from src to dst. Unlike standard memcpy, if the target address points to unallocated or protected space, it gracefully handles the page fault and returns the number of successfully read bytes instead of crashing. We use this strictly for runtime memory monitoring.

In this block, it safely reads 4 bytes from g_vmStart + 0x8a3 into g_vmContextAddr. 0x8a3 is our target's fixed offset within the .v-lizer section where the real context structure pointer (0x41E600) resides. Extracting this runtime pointer is what allows us to later correlate flat memory offsets to specific registers.

g_vmContext Allocation — Casts the dynamic address into our type-safe structural layout so we can safely view internal properties as they change during execution.

The exit hook (POPAD) is verified conditionally using Pin's conditional execution system:



C++

{

    INS_InsertIfCall(ins,   IPOINT_BEFORE, (AFUNPTR)CheckIfVmActive, IARG_END);

INS_InsertIfCall + INS_InsertThenCall — These work in tandem. The "If" routine (CheckIfVmActive) is called first. Conceptually, if PUSHAD marks the point where the Control Code initializes the virtual space by backing up the real CPU state, POPAD is the symmetric exit point where the virtual engine finishes its processing and restores those physical registers before returning to normal execution.

[-] Handler Entry Hooking — Tracking the Dispatcher Jump

The Main Handler dispatches execution to individual Opcode Handlers using an indirect memory jump: jmp [edi+eax*4]. We isolate this precise signature:



C++



if (INS_IsIndirectControlFlow(ins) && !INS_IsRet(ins) &&

    INS_IsValidForIpointTakenBranch(ins) &&

    INS_MemoryBaseReg(ins)  == REG_EDI &&

    INS_MemoryIndexReg(ins) == REG_EAX &&

    INS_MemoryScale(ins)    == 4)

{

    INS_InsertCall(ins, IPOINT_TAKEN_BRANCH, (AFUNPTR)OnHandlerEnter,

        IARG_REG_VALUE, REG_EAX,        // Decoded virtual opcode (Table index)

        IARG_BRANCH_TARGET_ADDR,        // Destination handler target address

        IARG_END);

}

Each clause forms a strict sieve to isolate the dispatcher jump:



INS_IsIndirectControlFlow — Verifies if the target branch location is dynamically calculated at runtime via registers or memory lookup (e.g., indirect JMP, CALL, or RET). This catches our target jmp [edi+eax*4].

!INS_IsRet — Because RET instructions fetch their targets dynamically from the stack, they also trigger INS_IsIndirectControlFlow. Since we are chasing the dispatcher jump and not standard function returns, we explicitly exclude returns.

INS_IsValidForIpointTakenBranch — Pre-checks if the instruction type safely supports an IPOINT_TAKEN_BRANCH analysis point. IPOINT_TAKEN_BRANCH evaluates states precisely at the moment a branch resolves to its new destination. Injects to non-branch instructions will throw critical JIT compiler exceptions inside Pin. This pre-check ensures we only hook valid branch paths.

Register Context Verification — Assures the memory operand structure matches [EDI + EAX*4]. Base must be REG_EDI, index must be REG_EAX, and the scale must equal 4 (DWORD array spacing). This isolates our dispatcher table index sequence from all other indirect branches.

The hook is attached to IPOINT_TAKEN_BRANCH to capture the exact transition into the target handler. Arguments passed are:



IARG_REG_VALUE, REG_EAX — Pass the value of EAX at the instant of the jump, which holds the decoded virtual opcode.

IARG_BRANCH_TARGET_ADDR — Pass the resolved destination address of the branch. Pin calculates the target address from [edi+eax*4] automatically and handles it cleanly.

INS_IsMemoryRead — Flags instructions retrieving data from memory.

The split between BEFORE and AFTER for writes is a vital engineering requirement:

The actual written data is extracted at IPOINT_AFTER by re-reading that saved address space. We verify INS_IsValidForIpointAfter because instructions lacking an unconditional fall-through path (like terminal branches or calls) cannot safely support an AFTER execution point.

[0x08] Analysis Routines and Region Classification

[-] Logging Handler Entry (OnHandlerEnter)

C++



VOID PIN_CLASS::OnHandlerEnter(ADDRINT opcode, ADDRINT target){

    g_curHandler = target;

    g_curOpcode  = opcode;

    g_inDispatch = false;              // Entering handler body -> Disable noise suppression



    HandlerInfo& h = g_handlerStats[target];

    h.callCount++;

    if (h.sampleOpcode.empty()) h.sampleOpcode = StringFromAddrint(opcode);



    LOG("[" + StringFromAddrint(g_step++) + "] op(DECODING)=" +

        StringFromAddrint(opcode) + " handler=" + StringFromAddrint(target) + "\n");

}

opcode / target — Receives our runtime values forwarded from IARG_REG_VALUE and IARG_BRANCH_TARGET_ADDR.

State Synchronization — Updates our tracker tracking variables, allowing sub-instruction memory reads and writes to know exactly which virtual handler owns them.

g_inDispatch = false — Since we have entered a true handler body, we disable noise suppression. This allows us to log operations until another dispatcher decode block is encountered. When decoding noise is caught, g_inDispatch = true mutes the output to clean up our log file (detailed in section [0x09]).

g_handlerStats — Indexes our statistics map using the handler address, updating callCount++ and tracking the first decoded opcode seen for this routine.

LOG — Emits the structured [0x..] op(DECODING)=.. handler=.. trace header to our log file, advancing g_step++.

[-] Automated Region Classification (RegionName)

This core helper decodes raw execution addresses into intelligible structural zones like virtual register slots, the stack, or bytecode zones.



C++



static std::string RegionName(ADDRINT x, ADDRINT esp, BOOL isStack){

    if (g_vmContextAddr && x >= g_vmContextAddr &&

        x < g_vmContextAddr + sizeof(VMCONTEXT))     // Within vm_context bounds?

    {

        UINT32 off = (UINT32)(x - g_vmContextAddr);  // Compute structure offset

        switch (off) {

        case 0x00: return "VM_EBP";  case 0x04: return "VM_EDI";

        case 0x08: return "VM_EBX";  case 0x0C: return "VM_EAX";

        case 0x10: return "VM_ESI";  case 0x14: return "VM_EDX";

        case 0x18: return "VM_ECX";  case 0x1C: return "VM_EFLAGS";

        case 0x30: return "init_flag";

        default:

            if (off >= 0x38) return "handler_table";

            return "VMCTX+" + StringFromAddrint(off);

        }

    }

    if (isStack) return "STACK";

    if (g_vmStart && x >= g_vmStart && x < g_vmEnd) return "Read OPCODE";

    return "";

}

Context Bounds Validation — If address x falls within [g_vmContextAddr, +sizeof(VMCONTEXT)), it belongs to the virtual VCPU context block.

Offset-to-Register Translation — Uses the delta to resolve exact named descriptors like VM_EAX or VM_EFLAGS. Offsets above 0x38 are labeled handler_table, while unconfirmed fields use a VMCTX+Offset syntax.

STACK Label — If the address sits outside the VCPU context but matches stack-relative instrumentation flags, it is designated as STACK.

Read OPCODE — Memory reads sourcing data directly from the .v-lizer code block are tagged as bytecode fetches.

Fallback — Unknown blocks return an empty token, which defaults to MEM (heap memory or target data space).

Tracking stack accesses (especially ESP-relative mutations) is vital. Virtual handlers use two primary strategies to handle execution state results: they either commit a value directly into a specific VM_Context register slot, or they push intermediate values to the stack (tracked via ESP) to feed the next handler. In the latter scenario, one handler stages a value on the stack, and its successor pops it. By logging these transfers as STACK, our tracer exposes the exact data pipeline flowing between discrete handlers.

Two formatting routines wrap this classification engine:



C++



static std::string Loc(ADDRINT addr, ADDRINT esp, BOOL isStack) {

    std::string r = RegionName(addr, esp, isStack);

    if (r.empty()) r = "MEM";

    return "[" + r + "]" + StringFromAddrint(addr);   // e.g., [VM_EAX]0x0041e60c

}static std::string ValTag(ADDRINT val, ADDRINT esp, BOOL isStack) {

    std::string r = RegionName(val, esp, isStack);

    return r.empty() ? "" : " (->" + r + ")";          // e.g., (->STACK)

}

Loc — Formats memory target logging to a uniform [Region]0xADDR style.

ValTag — If the value itself matches a tracked structural region address, it appends a helper pointer tag (e.g., (->STACK)). This makes pointer chaining across the stack extremely clear.

[-] Logging Memory Reads and LEA Computations

LEA calculations do not read memory; they calculate an effective address space. Thus, they emit calculated target locations without data values.



C++



VOID PIN_CLASS::OnLea(ADDRINT ea, ADDRINT esp, BOOL isStack){

    if (!g_curHandler || g_inDispatch) return;

    LOG("      LEA   -> " + Loc(ea, esp, isStack) + "\n");

}

ea — The dynamic calculated target forwarded via IARG_EXPLICIT_MEMORY_EA. Because no true memory transaction took place, no raw values are logged.

Loc(ea, ...) — Labels the calculated target (e.g., LEA -> [VM_EAX]0x0041e60c).

C++



VOID PIN_CLASS::OnMemRead(ADDRINT addr, UINT32 size, ADDRINT esp, BOOL isStack){

    if (!g_curHandler) return;

    g_handlerStats[g_curHandler].readCount++;

    if (g_inDispatch) return;



    ADDRINT val = 0;

    PIN_SafeCopy(&val, (VOID*)addr, (size < sizeof(ADDRINT)) ? size : sizeof(ADDRINT));

    LOG("      READ  " + Loc(addr, esp, isStack) +

        " val:" + StringFromAddrint(val) + ValTag(val, esp, isStack) + "\n");

}

Handler Verification — Skips instructions executing outside true handler boundaries (such as early dispatcher initialization).

readCount++ — Advances our handler statistics metrics. Note that LEA statements do not increment memory read counts because they are purely arithmetic operations.

Size Clamping — When invoking PIN_SafeCopy, we clamp the read bounds using (size < 4) ? size : 4. This prevents 1 or 2-byte narrow accesses from reading adjacent memory spaces and polluting our logs.

[-] Logging Memory Writes and Noise Filtering

The memory write prologue captures our target state:



C++



VOID PIN_CLASS::OnMemWriteBefore(ADDRINT addr, UINT32 size, ADDRINT esp){

    g_wAddr = addr; g_wSize = size; g_wEsp = esp;   // Values not committed yet; record coordinate targets

}

Our write epilogue performs value recovery alongside our noise-filtering layer:



C++



static const ADDRINT kDispatchMagic1 = 0x4b82;static const ADDRINT kDispatchMagic2 = 0x25c5;VOID PIN_CLASS::OnMemWriteAfter(BOOL isStack){

    if (!g_curHandler) return;

    g_handlerStats[g_curHandler].writeCount++;



    ADDRINT val = 0;

    PIN_SafeCopy(&val, (VOID*)g_wAddr, (g_wSize < sizeof(ADDRINT)) ? g_wSize : sizeof(ADDRINT));



    // If the committed data matches dispatcher signatures, mute logging until the next handler enter

    if (val == kDispatchMagic1 || val == kDispatchMagic2)

        g_inDispatch = true;

    if (g_inDispatch) return;



    LOG("      WRITE " + Loc(g_wAddr, g_wEsp, isStack) +

        " <- " + StringFromAddrint(val) + ValTag(val, g_wEsp, isStack) + "\n");

}

Post-Write Value Recovery — Reads our saved memory target coordinates (g_wAddr). Because the instruction has executed, we recover the true committed data.

Signature Detection — If the committed value matches 0x4b82 or 0x25c5, it signals that the dispatcher decoding engine has spun up. We set g_inDispatch = true to filter out these decoding instructions.

Noise Suppression — This flag automatically resets to false inside the next OnHandlerEnter callback. This filters out dispatcher noise while logging all critical handler execution details.
