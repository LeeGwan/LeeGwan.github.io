## 1. Why Code Virtualization?

A year ago, while studying system-level protection techniques, I ran into a whole family of anti-reversing technologies. Simple API detection or anti-debugging tricks already have well-known bypasses, so their defensive value is limited.

I eventually realized that the heart of anti-reversing is not "perfect blocking" but "stretching analysis time exponentially until the analyst gives up." Among those techniques, code virtualization is consistently cited as one of the hardest walls to break. This note collects what I learned and analyzed about how its basic structure works.

## 2. Structural Overview of a Virtualized Binary

If you pull apart a binary protected by a commercial protector, you quickly see that it does not run purely on the ordinary Intel instruction set we are used to. Internally it spins up its own Virtual Machine (VM) environment, and to do that it typically adds two new sections.

- **.VCPU section**: the engine itself — it builds the VM runtime and interprets/executes the virtual instructions.
- **.VCODE section**: a data region where the original assembly has been converted into a custom bytecode set that only the VCPU can read.

In the original code region (.text), the protected functions are left with nothing but an entry point into the VM (`jmp .VCPU`); the rest is NOP-ed out or overwritten with garbage. The moment control passes to the VM, a completely new set of rules applies.

### Full Flow: Real CPU → VCPU → VCODE

[![VM flow chart](/assets/img/vm_flowchart.png)](/assets/img/vm_flowchart.png)

*Full flow: Real CPU → VM Entry → Control Code → Main Handler → Opcode Handler*

As the flow chart shows, control that starts on the real CPU goes through the VM entry point into the Control Code, the Context region is initialized, and the Main Handler repeatedly reads bytecode from the VCODE region and dispatches it to handlers — an entirely new environment.

## 3. Dissecting the Virtual CPU Architecture

Different write-ups use slightly different names for the VM's parts. I'll organize it the way I intuitively understood it while following the code, splitting it into three pieces by role: the entry point, the main loop, and the instruction-processing routines.

First, one important premise. This VCPU is a **register machine**. That is, instead of pushing and popping values on a stack, it works by directly indexing fixed-slot virtual registers (REG_EAX, REG_EBP, REG_TMP, etc.) inside the context structure and reading/writing them. (If you follow a handler, you can see instructions directly manipulating named virtual registers like `mov REG_TMP, REG_EBP`. We confirm this in 3.3.)

### 3.1. Control Code (Entry and VM Setup)

This is the entry section that runs exactly once, the first time control passes from the real CPU into the VCPU environment.

[![Control Code disassembly](/assets/img/vm_control_code.png)](/assets/img/vm_control_code.png)

*Control Code disassembly and the real CPU register state right before entering the VM*

The disassembly above makes the Control Code's behavior clear.

It starts by backing up **to the stack** every physical register the real CPU was using (EAX–EDI) and the status flags (EFLAGS), via `pushad` and `pushfd`. The register pane on the right shows the real CPU state just before crossing into the VM (EAX=CCCCCCCC, EBP=0019FEE4, ESI=00411348, EFLAGS=00000216, etc.).

Those stack-backed values are then **copied** into the VM context region (0x41E600). These initial values serve two purposes:

- **Reference point for address math**: used by the operand handler for address arithmetic (e.g. computing `VM_EBP - 0x14`).
- **Initial values of the VM registers**: computation begins from this initial state.

Afterwards the instruction handlers **keep overwriting** these initial values with computed results, so the context slots reflect the live VM state in real time.

At the same time it sets up the Virtual Machine Context Address (the handler base) the VM will use at runtime (`add edi, 41E600`), and prepares the handler count plus the start address of the .VCODE section (in ESI) that holds the virtual instructions to execute. Here **ESI acts as the virtual program counter (VPC)** and becomes the reference for reading .VCODE in the main loop.

When initialization finishes, the final `jmp target.42122C` (Jmp Main Handler) hands control to the VM's main loop.

Comparing the state just before saving (context still all zeros) with the state right after the Control Code runs:

[![Context before save (all zeros)](/assets/img/vm_context_before.png)](/assets/img/vm_context_before.png)

*Before: the 0x41E600 context region is still empty (all 00)*

[![Context after save (values mapped)](/assets/img/vm_context_after.png)](/assets/img/vm_context_after.png)

*After: the real CPU register values backed up by pushad have been copied straight in*

If you dump the memory around 0x41E600, you can confirm that the real CPU register values just backed up by `pushad` **have been copied straight in**. Each slot of this context region becomes an **independent VM register**.

**Roles of the context slots:**

- **Initial stage**: VM_EAX=CCCCCCCC, VM_EBP=0019FEE4… (copied from the CPU's initial values)
  - *Why? The VM enters in the middle of a function, so the original function's register state is needed as a reference point.*
- **Execution stage**: the operand handler reads the initial values to compute an address/value → stores it on the stack
  - *e.g. using the initial value in an address computation like VM_EBP(0019FEE4) - 0x14*
- **Store stage**: the instruction handler takes the stack value and stores it into a context slot (overwriting the initial value)
  - *the result is stored into the context, reflecting the VM's latest state*
- **Repeat**: every handler keeps updating context values to reflect the VM state
  - *Why this complex structure is needed, and exactly how each handler works, is covered in detail in the next part (Handler analysis).*

The VCPU is now ready to process virtual instructions based on the slots in this context space.

For reference, dumping this context region in 4-byte (DWORD) units looked roughly like this (based on the register values just before saving):

| Offset | Dump (little-endian) | Decoded | Register |
|--------|----------------------|---------|----------|
| +0x00 | E4 FE 19 00 | 0019FEE4 | EBP |
| +0x04 | E4 FE 19 00 | 0019FEE4 | EDI |
| +0x08 | 00 B0 2D 00 | 002DB000 | EBX |
| +0x0C | CC CC CC CC | CCCCCCCC | EAX |
| +0x10 | 48 13 41 00 | 00411348 | ESI |
| +0x14 | 28 7B 53 00 | 00537B28 | EDX |
| +0x18 | 00 00 00 00 | 00000000 | ECX |
| +0x1C | 16 02 00 00 | 00000216 | EFLAGS |

The save order differs from the native `pushad` order (EAX, ECX, EDX, EBX, ESP, EBP, ESI, EDI) because the VCPU follows its own register layout (a deliberate shuffling of the order).

*.VCODE region: data encoded as custom bytecode rather than normal assembly*

This is a memory dump of the .VCODE region the Control Code set up. Rather than ordinary assembly, it is encoded as a custom bytecode form. Having finished basic setup, the VM now moves into this region and begins reading data.

### 3.2. Main Handler (VM Dispatcher)

This is the main loop entered after the Control Code has set up the VM environment. To process virtual instructions it repeats Fetch – Decode – Execute endlessly.

[![Main Handler CFG](/assets/img/vm_main_handler.png)](/assets/img/vm_main_handler.png)

*Main Handler's Fetch – Decode – Execute control-flow graph*

The control-flow graph (CFG) above clearly shows how the Main Handler fetches and decodes a virtual instruction.

**Fetch Instruction**: first it uses `lodsb` to read one byte of virtual opcode from the .VCODE section into AL. Since `lodsb` reads from the location ESI points to and auto-increments ESI by one, this confirms that **ESI is the virtual program counter (VPC)**.

**Decode Instruction**: the fetched value is not used as-is. The following instructions run through several arithmetic steps such as `sub al, bl` and `add al, bh` to decrypt the encrypted opcode. The interspersed `[esp]`-manipulating code like `push`, `add esp, 2`, and `mov [esp], bx` are not virtual-register operations — they are **temporary scratch math for the decryption plus obfuscation/junk code meant to hamper analysis**. Finally, the `movzx eax, al` annotated "Finish DECODE Instruction" stores the fully decrypted index value into EAX.

**Execute Instruction**: the last block executes `jmp dword ptr ds:[edi+eax*4]`. Here EDI is the base address of the OPCODE handler table and EAX is the instruction index we just decrypted. So this jumps to the address of the handler that processes that virtual instruction. This pattern — indexing a table to branch to a handler — is exactly what shows this VCPU to be a register-machine dispatcher.

### 3.3. OPCODE Handlers

These are the routines branched to from the Main Handler's Execute step that actually process the individual virtual instructions (mov, add, call, etc.).

The most striking thing during the research was how simple the custom VCPU's memory addressing is. For example, it cannot handle an indexed instruction like `mov [ebp-0x14], ebx` in one shot; it splits it like this:

- `mov REG_TMP, REG_EBP` (copy virtual EBP into a temporary virtual register)
- `sub REG_TMP, 0x14` (offset arithmetic)
- `mov [REG_TMP], REG_EBX` (store the value at the final computed address)

Here REG_TMP, REG_EBP, and REG_EBX are all **fixed virtual-register slots** inside the 0x41E600 context we saw earlier. They are exactly the values the Control Code backed up to the stack and copied into the context, and from then on every handler reads and writes these slots directly. Using fixed memory slots as virtual registers and directly indexing named registers to read/write is precisely **the hallmark of a register-machine architecture**.

#### How Operand Handlers and Instruction Handlers Cooperate

In practice the handlers go through the stack to reach these context slots:

1. **Operand Handler** (prepares a value/address)
   - **Read initial value**: read something like VM_EBP from the context
   - **Compute address**: operate on that value (e.g. `VM_EBP - 0x14` → a memory address)
   - `mov dword ptr [esp], eax` : store the result on the stack
2. **Instruction Handler** (actual operation and store)
   - `pop eax` : read the address/value from the stack
   - perform a memory access or further computation
   - `mov dword ptr [edx], eax` : **store the new result into the context ([edx])** ← overwriting the initial value

In other words, **the initially copied CPU values are used as reference points for address math, and the instruction handlers keep overwriting them with computed results.** The stack is just a temporary conduit; the **real data/state lives in the context slots** and is continuously updated.

Because a single simple operation is replaced by several virtual instructions, virtualized code inevitably balloons in size compared to the original. This is the decisive reason the protector cannot overwrite the original code in place and instead creates a separate .VCODE section.

One pattern stands out in this log. In the early part ([0]–[27]), the three handlers `0x41f566 → 0x421482 → 0x4202e0` keep repeating in groups of three.

Their roles:

- `0x41f566`: Operand Handler — reads bytecode from .VCODE and pushes a value onto the stack
- `0x421482`: Operand Handler — reads or computes additional info and pushes it onto the stack
- `0x4202e0`: Instruction Handler — takes the stack value and actually stores/operates it into a VM register slot in the context

This shows that even a simple original action (e.g. storing a constant into a virtual register) is split into multiple operand-handler + instruction-handler steps. The earlier point — "one simple operation balloons into many virtual instructions" — shows up directly in the trace.

The last line, `End Result= 0x12345678`, is the final value this virtualized code fragment computed after going through all the handlers. That value will be sitting in some VM register slot of the context (around 0x41E600), e.g. VM_EAX or VM_EDX. In other words, an operation that looks very short to a human actually took more than 70 handler calls to compute and store a single result into a context slot.
