## 1. Starting the Study: Why Code Virtualization?

A year ago, while studying system-level protection techniques, I came across various technologies designed to defend against reversing. Simple API detection or anti-debugging techniques already have many widely known bypass methods, showing clear limitations in defense.

Ultimately, I realized that the core of anti-reversing is not 'perfect blocking' but 'exponentially increasing the time required for analysis to make the attacker give up.' Among these, Code Virtualization is considered the most difficult wall to penetrate. I have organized my study and analysis notes on how it operates and its basic structure.

## 2. Structural Understanding of a Code Virtualization Binary

When you tear apart a virtualized binary applied with a commercial protector, you can see that it doesn't operate solely on the standard Intel CPU instruction set we know. Internally, it takes a structure that adds two main new sections to boot up its own Virtual Machine environment.

- **.VCPU Section**: The engine body that builds the runtime environment of the virtual machine and interprets and executes virtual instructions.
- **.VCODE Section**: The data area where the original assembly code is converted and stored as a custom instruction set (Bytecode) that only the VCPU can read.

The protected target functions in the original code area (.text) are either NOPed out or overwritten with garbage values, leaving only the entry point (`jmp .VCPU`) to enter the virtual machine. From the moment control is handed over to the virtual machine, completely new rules apply.

### Overall Flow: Real CPU → VCPU → VCODE

[![VM Overall Flowchart](/assets/img/vm_flowchart.png)](/assets/img/vm_flowchart.png)

*Overall Flow: Real CPU → VM Entry → Control Code → Main Handler → Opcode Handler*

As seen in the flowchart above, the control flow starting from the actual CPU passes through the VM Entry Point and enters the Control Code. The Context area is initialized, and a completely new environment is established where the Main Handler repeatedly reads the bytecode in the VCODE area and dispatches it to the handlers.

## 3. Digging into the Virtual CPU Architecture

Although the terms used to refer to the virtual machine structure differ slightly in each analysis document, I have divided and organized them into three main elements based on their roles, according to how I intuitively understood them while tracing the code: Control Code, Main handler, and OPCODE Handlers.

First, let's point out one important premise. This VCPU is a **Register Machine** structure. In other words, instead of pushing and popping values on a stack, it operates by directly indexing, reading, and writing to virtual registers (REG_EAX, REG_EBP, REG_TMP, etc.) that have fixed positions within the context structure. (Actually, if you follow the handlers, you can see instructions directly manipulating named virtual registers like `mov REG_TMP, REG_EBP`. We will confirm this in section 3.3 later.)

### 3.1. Control Code (Entry and Virtual Environment Setup)

This is the entry part that is executed exactly once first when control is transferred from the Real CPU environment to the VCPU environment.

[![Control Code Disassembly](/assets/img/vm_control_code.png)](/assets/img/vm_control_code.png)
*Control Code disassembly and Real CPU register state right before entry*

Looking at the disassembly screen above, you can clearly see the actual operation process of the Control Code.

First of all, it starts by **backing up to the stack** all the Registers (EAX~EDI) and status flags (EFLAGS) used by the Real CPU through the `pushad` and `pushfd` instructions. Looking at the register window on the right, you can check the actual CPU state values right before crossing over to the virtual environment (EAX=CCCCCCCC, EBP=0019FEE4, ESI=00411348, EFLAGS=00000216, etc.).

Next, these values backed up on the stack are **copied** to the Virtual Machine Context (0x41E600) area. These initial values serve two roles:

- **Reference point for address calculation**: Used when the Operand Handler performs address calculations (e.g., calculating `VM_EBP - 0x14`).
- **Initial values for VM registers**: Operations begin based on the initial state.

Afterward, since the Instruction Handlers **continuously overwrite** these initial values with operation results, the slots in the Context reflect the VM state in real-time.

At the same time, it sets up the Virtual Machine Context Address (handler base address) that the virtual machine will use at runtime (`add edi, 41E600`), and prepares the number of handlers and the start address of the .VCODE section, which contains the virtual instructions to be executed, into a register (ESI). Here, **ESI acts as VM_EIP**, and becomes the standard for reading .VCODE in the subsequent main loop.

When initialization is complete, control is handed over to the main loop of the virtual machine in earnest through `jmp target.42122C` (Jmp Main Handler) at the final part.

Comparing the state right before saving (where the context is entirely empty with 00) and the state right after the Control Code execution where values are filled, it looks like the following:

[![Context before save (all 00)](/assets/img/vm_context_before.png)](/assets/img/vm_context_before.png)

*Before save: The 0x41E600 context area is completely empty with 00*

[![Context after save (value mapping)](/assets/img/vm_context_after.png)](/assets/img/vm_context_after.png)

*After save: The Real CPU register values backed up by pushad are copied exactly as they are*

If you dump the memory of the 0x41E600 range, you can confirm that **the Real CPU register values previously backed up to the stack with `pushad` have been copied exactly as they are**. Each slot in this Context area becomes an **independent register of the VM**.

**Roles of Context Slots:**

- **Initial Phase**: VM_EAX=CCCCCCCC, VM_EBP=0019FEE4... (Copied as CPU initial values)
  - *Why?: Since the VM is entered in the middle of a function, the register state of the original function is needed as a reference point.*
- **Execution Phase**: The Operand Handler reads the initial values to calculate addresses/values → saves to the stack.
  - *Example: Uses initial values for address calculations like VM_EBP(0019FEE4) - 0x14.*
- **Saving Phase**: The Instruction Handler retrieves stack values and saves them to Context slots (overwriting initial values).
  - *The result value is saved to the Context to reflect the latest VM state.*
- **Repetition**: For every handler, Context values are continuously updated, reflecting the VM state.
  - *Why this complex structure is necessary and how each handler operates specifically will be covered in detail in the next chapter (Handler Analysis).*

Now, the VCPU has finished preparing to process virtual instructions based on the slots in this Context space.

### 3.2. Main Handler (VM Dispatcher)

This is the main loop entered after the virtual environment is set up by the Control Code. It infinitely repeats the Fetch - Decode - Execute steps to process virtual instructions.

[![Main Handler CFG](/assets/img/vm_main_handler.png)](/assets/img/vm_main_handler.png)

*Fetch - Decode - Execute of the Main Handler*

Looking at the Control Flow Graph (CFG) above, the flow of how the Main Handler fetches and decodes virtual instructions is clearly revealed.

**Fetch Instruction**: First of all, it uses the `lodsb` instruction to read 1 byte of the virtual Opcode from the .VCODE section and places it in the AL register. `lodsb` reads from the location pointed to by ESI and automatically increments ESI by 1 (here, ESI is VM_EIP).

**Decode Instruction**: It does not use the fetched value as is. Looking at the subsequent instructions, it goes through a decryption process of the encrypted Opcode via several stages of arithmetic operations like `sub al, bl`, `add al, bh`, etc. The `[esp]` manipulation codes mixed in between, such as `push`, `add esp, 2`, and `mov [esp], bx`, are not operations handling virtual registers, but rather **temporary calculations for the decryption process + Obfuscation and junk code to hinder analysis**. Finally, through `movzx eax, al` tagged with the "Finish DECODE Instruction" comment, the fully decrypted index value is stored in the EAX register.

**Execute Instruction**: In the last block, it executes the `jmp dword ptr ds:[edi+eax*4]` instruction. Here, EDI is the base address of the OPCODE Handlers Table, and EAX is the instruction index just decrypted. In other words, through this operation, it jumps to the handler address that will actually process the corresponding virtual instruction and directs the operation. This structure of referencing the table with an index and branching to a handler is exactly what shows that this VCPU is a Register Machine-based dispatcher.

### 3.3. OPCODE Handlers

These are the routines that actually process individual virtual instructions (mov, add, call, etc.) branched through the Execute step of the Main Handler.

The most impressive point during the research process was that the memory addressing method of the custom-defined VCPU is extremely simple. For example, it cannot process an index operation instruction like `mov [ebp-0x14], ebx` at once and breaks it down to process as follows:

- `mov REG_TMP, REG_EBP` (Copy virtual EBP to a temporary virtual register)
- `sub REG_TMP, 0x14` (Offset calculation)
- `mov [REG_TMP], REG_EBX` (Insert value into the final calculated address)

Here, REG_TMP, REG_EBP, and REG_EBX are all **virtual register slots with fixed positions** within the 0x41E600 Context seen earlier. The values copied to the Context after backing up CPU registers to the stack in the Control Code are precisely these slots, and all subsequent handlers directly read, write, and calculate with these slots. Using memory slots at fixed locations as virtual registers and directly indexing, reading, and writing to named registers is exactly the **characteristic of a Register Machine structure**.

#### Cooperative Structure of Operand Handler and Instruction Handler

In reality, handlers go through the stack to access these Context slots:

1. **Operand Handler** (Prepare Value/Address)
   - **Read Initial Value**: Reads a value like VM_EBP from the Context.
   - **Calculate Address**: Performs calculations based on that value (e.g., `VM_EBP - 0x14` → Calculate memory address).
   - `mov dword ptr [esp], eax` : Saves the calculation result to the stack.
2. **Instruction Handler** (Actual Operation and Saving)
   - `pop eax` : Reads the address/value from the stack.
   - Performs memory access or additional calculations.
   - `mov dword ptr [edx], eax` : **Saves the new result value to the Context ([edx])** ← Overwrites the initial value.

In other words, **the initially copied CPU values are used as a reference point for address calculation, and then the Instruction Handler continuously overwrites them with the operation results.** The stack is a temporary pathway, and **the actual data/state is stored in the slots of the Context** and is continuously updated.

Because a single simple operation is replaced by multiple virtual instructions, the file size of the code with virtualization applied inevitably expands significantly compared to the original. This was the decisive reason for creating a separate section named .VCODE instead of just overwriting the original code.

One pattern stands out in this log. Looking at the early part ([0]~[27]), you can see that three handlers `0x41f566 → 0x421482 → 0x4202e0` continue to repeat in groups of three.

Looking at the role of each handler:

- `0x41f566`: Operand Handler — Reads bytecode from .VCODE and saves the value to the stack
- `0x421482`: Operand Handler — Reads or calculates additional information and saves it to the stack
- `0x4202e0`: Instruction Handler — Fetches the value from the stack and actually saves/operates it into the VM register slot in the Context

This shows that even a simple operation of the original (e.g., saving a constant to a virtual register) is broken down into multiple stages of operand handler + instruction handler for processing. The point mentioned earlier that "a single simple operation expands into multiple virtual instructions" is revealed exactly as is in the trace.

The `End Result= 0x12345678` in the last line is the final computed result value after this virtualized piece of code has passed through all the handlers. This value will be stored in a VM register slot (e.g., VM_EAX, VM_EDX) within the Context (0x41E600 range). In other words, an original operation that seems very short to humans actually went through over 14 handler calls to save a single result into a Context slot.

To process such short code, numerous handlers were mobilized to produce a result. Next time, we will find out step by step what specific roles each handler plays.
