## [0x00] Overview

In [Part 1](https://leegwan.github.io/posts/2026-05-25-oreans-vm-1/) we saw the big picture: a virtualized binary is split into `.VCPU` / `.VCODE`, and the control flow circulates through Control Code → Main Handler → Opcode Handler. And at the end, I just threw up the handler trace log and `End Result = 0x12345678`, putting off "what each handler does, next time."

This post is that next time. I followed the handlers found in the trace one by one with a debugger, watching how the virtualized code actually rolls along inside the handlers. Honestly, when I first just looked at the trace, I had no idea what those 72 lines of handler calls were even doing, but after following it all the way through, the conclusion was almost disappointingly simple. These 72 steps (14 distinct handlers appear) ended up being just one line of the original.

```asm
mov [ebp-8], 0x12345678
; afterwards it returns to 0x411919 and uses the [ebp-8] value as a function argument
```

To the human eye it's one line, but inside the VM 14 handlers get called 72 times to produce this one line. This time I directly confirmed exactly how "a single simple operation expands into dozens of steps," which I mentioned in Part 1, happens at the handler level.

 **Note:** The values used in the analysis (addresses, registers, etc.) differ from debugging session to session. The point is not the absolute addresses but what the handlers do, so please treat the captured values only as a reference for following the flow.

## [0x01] Two Premises for Reading the Handlers

Before cracking open the handlers, let me first sort out just two things. Knowing these makes the 14 handlers come into view much faster.

**[-] The stack is the conduit between handlers**

As we saw in Part 1, this VCPU is a register machine, and the actual state lives in VM_Context slots. But when a handler accesses those slots, it always goes through the stack once. When one handler finishes and leaves an address or value at `[esp]`, the next handler receives it and does its own job. So in most cases two or three handlers have to come together before one meaningful operation is complete.

**[-] Almost every handler has junk mixed in**

Open up a handler and you'll find tons of code that does `sub esp,4` and then undoes it, briefly borrows `ebp`/`ebx` and restores them, and inserts `not`/`and`/`xor`/`xchg` operations and then throws away the result. It's all obfuscation meant to hinder analysis. Below, I only write the **actual behavior** with this junk stripped away.

## [0x02] The VM_Context Structure

Let me first pin down the VM_Context that the handlers commonly reference, then move on. The base is `0x0041E600`, and at runtime EDI points to this address. In Part 1 I only organized up to the register slots (+0x00 ~ +0x1C), but as I followed the handlers I was able to fill in the area after that and the location of the handler table. The actual dump is this.

```
0041E600  E4 FE 19 00  E4 FE 19 00  00 30 38 00  CC CC CC CC
0041E610  48 13 41 00  88 7B 69 00  00 00 00 00  16 02 00 00
0041E620  00 00 00 00  00 00 00 00  06 00 00 00  00 00 00 00
0041E630  01 00 00 00  00 00 00 00  0E 0D 42 00  88 F6 41 00
```

Interpreted in DWORD units, it's as follows.

| Offset | Field | Captured Value | Notes |
|--------|------|---------|------|
| +0x00 | VM_EBP | 0019FEE4 | |
| +0x04 | VM_EDI | 0019FEE4 | |
| +0x08 | VM_EBX | 00383000 | |
| +0x0C | VM_EAX | CCCCCCCC | |
| +0x10 | VM_ESI | 00411348 | |
| +0x14 | VM_EDX | 00697B88 | |
| +0x18 | VM_ECX | 00000000 | |
| +0x1C | VM_EFLAGS | 00000216 | The slot the trio fills first |
| +0x20 | ? | 0 | Purpose not confirmed |
| +0x24 | ? | 0 | Purpose not confirmed |
| +0x28 | ? | 6 | A value the `[15]` handler writes; couldn't find where it's read afterwards |
| +0x2C | ? | 0 | A value the `[42]` handler adds to the return value |
| +0x30 | init_flag | 1 | Target of the Control Code `[edi+30]` lock cmpxchg |
| +0x34 | ? | 0 | Boundary right before the handler table |
| +0x38~ | handler_table[] | 0x420D0E, 0x41F688, … | Array of Opcode handler addresses |

```c
// base = 0x0041E600 (at runtime EDI points to it)
struct VM_CONTEXT {
    uint32_t VM_EBP;        // +0x00
    uint32_t VM_EDI;        // +0x04
    uint32_t VM_EBX;        // +0x08
    uint32_t VM_EAX;        // +0x0C
    uint32_t VM_ESI;        // +0x10
    uint32_t VM_EDX;        // +0x14
    uint32_t VM_ECX;        // +0x18
    uint32_t VM_EFLAGS;     // +0x1C

    uint32_t unk_20;        // +0x20  unknown
    uint32_t unk_24;        // +0x24  unknown
    uint32_t unk_28;        // +0x28  [15] writes 6, no access afterwards
    uint32_t unk_2C;        // +0x2C  [42] adds to return value (captured value 0 this time)
    uint32_t init_flag;     // +0x30  Control Code lock cmpxchg target
    uint32_t unk_34;        // +0x34  unknown

    void*    handler_table[]; // +0x38~  Main Handler branches via jmp [edi+eax*4]
};
```

The handler table starts at `+0x38`. Looking at the dump, address values like `+0x38 = 0x00420D0E`, `+0x3C = 0x0041F688` are stored, and the Main Handler indexes this table with `jmp [edi+eax*4]` to branch to a handler.

## [0x03] Handler Catalog

The 72 trace steps are all combinations of the 14 handlers below. I only write the actual behavior with the junk stripped away.

| Handler | Index (EAX) | Actual Behavior |
|--------|-------------|----------|
| `0x41f566` | 0x2d | Reads 1 byte from `.VCODE` and decodes it, computes the VM register slot address with `lea eax,[edi+eax*4]` and pushes it onto the stack |
| `0x421482` | 0x89 | `mov edx,[esp]` — the address/value on the stack into edx |
| `0x4202e0` | 0x32 | `pop dword ptr ds:[edx]` — stores the value on the stack into the target address |
| `0x4202ce` | 0x58 | Writes 6 to VM_Context `+0x28` (no access afterwards) |
| `0x41ec7e` | 0x76 | `mov edx, esp` — captures the current stack address |
| `0x41fae3` | 0x9b | `mov [esp], edx` — loads the address/value onto the stack |
| `0x41fb68` | 0x5e | Reads 4 bytes from `.VCODE` with `lodsd`, decrypts via `xor`+`add` accumulation, then `mov [esp],eax` |
| `0x41fdb4` | 0x1d | `mov eax,[esp]` + correction via esi + `add [esp],eax` — base + offset address computation |
| `0x4218bf` | 0x75 | `mov esp,[esp]` — switches the working esp to the computed address |
| `0x4207e6` | 0x33 | `push dword ptr ds:[edi+eax*4]` — pushes a VM_Context slot value onto the stack |
| `0x4208b2` | 0x94 | return value `+= [VM_Context+0x2C]` |
| `0x41ee2a` | 0x7a | `lodsd` + `add esi,eax` — adds the read value to ESI |
| `0x41fb23` | 0x25 | `push dword ptr ds:[edx]` — pushes the value the slot points to onto the stack |
| `0x41f6a6` | 0x1a | Stack cleanup + `popad`/`popfd` + `ret` — restores the Real CPU and returns |

When you cut the trace into handler patterns, it divides into five sections. I'll go through them in order starting from 0x04.

| Section | Trace | What it does |
|------|----------|---------|
| Phase 1 | `[0]~[27]` | Saves the backed-up registers into VM_Context slots |
| Phase 2 | `[28]~[40]` | Builds the `[ebp-8]` address and writes the decrypted `0x12345678` |
| Phase 3 | `[41]~[43]` | Decrypts/sets the return address, changes ESI with `add esi,eax` |
| Phase 4 | `[44]~[70]` | Restores VM_Context slot values to the stack |
| Phase 5 | `[71]` | Returns via `popad`/`popfd` + `ret` |

## [0x04] Phase 1 — Filling VM_Context with Registers

This is the `0x41f566 → 0x421482 → 0x4202e0` trio section that caught the eye first in the Part 1 trace. One cycle is precisely "save one value into a VM_Context slot."

### [-] Following the first trio

Right after entry, a backed-up register value (the first cycle is EFLAGS=0x216) is on top of the stack, and the trio moves it into a slot.

![0x41f566](/assets/img/Code Virtualization/Code Virtualization-2/part1/0x41f566.png)

`0x41f566` reads 1 byte with `lodsb`, decodes it with `sub al,bl`, then makes an index with `movzx eax,al`, computes the slot address with `lea eax,[edi+eax*4]`, and pushes it onto the stack with `mov [esp],eax`. The first cycle result is EAX = 0x0041E61C (base + 0x1C, i.e. the EFLAGS slot), and `0019FE04 = 0041E61C` is left on the stack.

![0x421482](/assets/img/Code Virtualization/Code Virtualization-2/part1/0x421482.png)

`0x421482` puts that slot address into edx with `mov edx,[esp]` (EDX = 0041E61C).

![0x4202e0](/assets/img/Code Virtualization/Code Virtualization-2/part1/0x4202e0.png)

`0x4202e0` writes the value on the stack into the slot with `pop dword ptr ds:[edx]`. At the bottom of the debugger `[edx]=[0041E61C]=216` is shown, and `0041E61C` in the dump gets filled with `16 02 00 00` (=0x216).

Combining the three handlers gives this one line.

```asm
mov [VM_Context+0x1C], 0x216   ; save the backed-up EFLAGS value into the slot
```

`0x41f566` is **where** (the slot address), `0x421482` fixes that address as the working register, and `0x4202e0` finally writes **what** (the stack value) to that address. It's exactly the "prepare address → fix address → write value" three-beat seen in the Part 1 OPCODE Handlers.

### [-] The 9 repetitions

This trio repeats 9 times across `[0]~[27]`, filling the register values backed up by the Control Code into the slots one by one. Only the `[15]` `0x4202ce` in the middle breaks out of the trio — it writes 6 to VM_Context `+0x28`, but that value isn't read anywhere afterwards. Honestly I can't tell why it's there from this trace alone, so I'll leave it as unknown.

## [0x05] Phase 2 — mov [ebp-8], 0x12345678

This is the most important section in this post. It produces both the **value (0x12345678)** to write and the **address ([ebp-8])** to write to, decrypting both from `.VCODE`.

### [-] Making the value and putting it down temporarily ([28]~[33])

![0x41ec7e](/assets/img/Code Virtualization/Code Virtualization-2/part2/0x41ec7e.png)

`[28]` `0x41ec7e` — `sub esp,4`·`mov [esp],esp`·`add [esp],4`·`mov edx,[esp]`·`add esp,4`. Simply put, it puts the current stack address into edx (`mov edx, esp`). Result EDX = 0019FE08.

![0x41fae3](/assets/img/Code Virtualization/Code Virtualization-2/part2/0x41fae3.png)

`[29]` `0x41fae3` — puts that address back onto the stack with `mov [esp], edx` (`[0019FE04] = 0019FE08`).

![0x41fb68](/assets/img/Code Virtualization/Code Virtualization-2/part2/0x41fb68.png)

`[30]` `0x41fb68` — reads 4 bytes from `.VCODE` with `lodsd`, does `xor eax,ebx`, then accumulates with several `add` operations to decrypt. This cycle's result is 0x00000004 (`0019FE00 = 00000004`).

`[31]` `0x41fdb4` — receives the offset with `mov eax,[esp]` (=4), corrects it via esi, then adds the offset to the base with `add [esp],eax` to compute the `esp+4` position.

![0x4218bf](/assets/img/Code Virtualization/Code Virtualization-2/part2/0x4218bf.png)

`[32]` `0x4218bf` — switches the working esp to that address with `mov esp, dword ptr ss:[esp]`.

![0x41fb68 value decryption](/assets/img/Code Virtualization/Code Virtualization-2/part2/0x41fb68_값복호화.png)

`[33]` `0x41fb68` — the same decoder is called again, and this time the value that had been encrypted is decrypted into 0x12345678 (`EAX = 12345678`). However, the 0019FE08 at this point is not the final destination but an intermediate scratch.

### [-] Computing the real address and storing ([34]~[40])

![0x4207e6](/assets/img/Code Virtualization/Code Virtualization-2/part3/0x4207e6.png)

`[34]` `0x4207e6` — pushes the EBP value stored in VM_Context onto the stack with `push dword ptr ds:[edi+eax*4]`. Not using the register directly but pulling it out of the context is exactly the register-machine structure seen in Part 1.

`[35]` `0x421482` → `mov edx,[esp]`, `[36]` `0x41fae3` → `mov [esp],edx` move the value.

`[37]` `0x41fb68` — the decoder is called again, and decrypting the value read from `.VCODE` gives 0xFFFFFFF8, that is **-8** (`0019FE00 = FFFFFFF8`).

![0x41fdb4](/assets/img/Code Virtualization/Code Virtualization-2/part3/0x41fdb4.png)

`[38]` `0x41fdb4` — adds `mov eax,[esp]` (=0xFFFFFFF8) and EBP. `0019FEE4(EBP) + (-8) = 0019FEDC`, which is the actual address of `[ebp-8]`.

`[39]` `0x421482` — brings that address (0019FEDC) into edx.

![0x4202e0](/assets/img/Code Virtualization/Code Virtualization-2/part3/0x4202e0.png)

`[40]` `0x4202e0` — writes the value on the stack to that address with `pop dword ptr ds:[edx]`. EDX = 0019FEDC, the popped value = 0x12345678. `0019FEDC` in the dump gets filled with `12345678`.

To summarize, `[28]~[39]` build the EBP-8 (= 0019FEDC) address, and `[40]` writes 0x12345678 there.

```asm
mov [ebp-8], 0x12345678
```

There's something worth pointing out here. Not only the value to store (0x12345678) but **even the address offset (-8) is not left in plaintext** — it's encrypted in `.VCODE` and unpacked at runtime with `0x41fb68` to build `[ebp-8]`. Both the constant and the offset only reveal themselves after going through the decoder handler. This is the point that makes static analysis annoying.

## [0x06] Phase 3 — Return Preparation

![0x41fb68 return address](/assets/img/Code Virtualization/Code Virtualization-2/part4/0x41fb68.png)

`[41]` `0x41fb68` — the decoder this time decrypts the return address 0x411919 and sets it on the stack (`0019FE08 = 00411919`). Looking at the code region together, 0x411919 has `mov eax, dword ptr ss:[ebp-8]`, and the debugger even annotates it as "ret address." In other words, when you exit the virtualization block you return to this address and re-read the `[ebp-8] = 0x12345678` just written.

![0x4208b2](/assets/img/Code Virtualization/Code Virtualization-2/part4/0x4208b2.png)

`[42]` `0x4208b2` — grabs the VM_Context `+0x2C` address with `edx = edi+0x2C`, reads that value and adds it to the return address (debugger annotation `read vm_context+2c`). The actual behavior is `[return address] += [VM_Context+0x2C]`. However, in this capture `+0x2C = 0`, so 0x411919 stays the same.

 **Note:** I can't conclude from this single trace what kind of correction it's trying to make on a path where `+0x2C` is non-zero. For now I'll only record "in this case the added value is 0, so it's unchanged." That said, personally I speculate it might be a slot for adding the delta to the return address to correct it when ASLR has changed the image base (this is purely speculation).

![0x41ee2a](/assets/img/Code Virtualization/Code Virtualization-2/part4/0x41ee2a.png)

`[43]` `0x41ee2a` — reads a value from `.VCODE` with `lodsd` (`01 00 00 00` → 1) and adds that value to ESI with `add esi,eax`. The `mov ebx`·`not`·`and`·`xor`·`push`/`pop edi` seen alongside are junk whose results aren't used.

## [0x07] Phase 4 — Returning VM_Context Back to the Stack

This section's trio is `0x41f566 → 0x421482 → 0x41fb23`. The first two are the same as Phase 1, and **only the last** changes from `0x4202e0` (store) to `0x41fb23` (read and push).

![0x41fb23](/assets/img/Code Virtualization/Code Virtualization-2/part5/0x41fb23.png)

Looking at the first cycle, `[44]` `0x41f566` puts the EFLAGS slot address (0x0041E61C) onto the stack, `[45]` `0x421482` puts it into edx (EDX = 0041E61C), and `[46]` `0x41fb23` pushes the slot value onto the stack with `push dword ptr ds:[edx]`. At the bottom `[edx]=[0041E61C]=216` is shown, and the EFLAGS value piles onto the stack.

It's exactly the opposite direction of Phase 1. If Phase 1 was "stack → slot (store)," here it's "slot → stack (restore)." This trio repeats 9 times across `[44]~[70]`.

![almost-final stack state](/assets/img/Code Virtualization/Code Virtualization-2/part5/거의막바지.png)

Looking at the stack, the return address set in `[41]` (0x411919) is on top, and below it the register values restored by `[44]~[70]` (EFLAGS=0x216, EBP=0019FEE4, etc.) pile up neatly. In the end it's the work of reconstructing the registers backed up in VM_Context back onto the stack before exiting the virtualization block, and this is preparation for the `popad`/`popfd` that follows right after.

## [0x08] Phase 5 — Exiting via popad/popfd

`[71]` `0x41f6a6` — after cleaning up the stack, it does this.

```asm
sub dword ptr ds:[edi+30], 6AE5   ; VM_Context +0x30 slot
popad                             ; restore the 8 general registers piled up by [44]~[70]
popfd                             ; restore EFLAGS
ret                               ; return to the return address at the top of the stack (0x411919)
```

It restores the Real CPU registers and flags at once with `popad`/`popfd`, and returns to 0x411919 with `ret`. It's the exit symmetric to the `pushad`/`pushfd` entry of the Part 1 Control Code. Before that, with `sub dword ptr ds:[edi+30], 6AE5` it also touches the `+0x30` slot once.

Right after returning, EIP points to 0x411919, and the `mov eax, dword ptr ss:[ebp-8]` at that spot reads the `[ebp-8] = 0x12345678` just written. Then, with the `push 417B30` ("%d") → `call 411046` flow, it passes that value as a function argument.

## [0x09] Full Flow Summary

| Section | Trace | Summary |
|------|----------|------|
| Phase 1 | `[0]~[27]` | Save backed-up registers into VM_Context slots (trio `0x41f566 → 0x421482 → 0x4202e0` ×9, `[15]` writes 6 to `+0x28`) |
| Phase 2 | `[28]~[40]` | Decrypt the value (0x12345678) and offset (-8), store at `[ebp-8]` → `mov [ebp-8], 0x12345678` |
| Phase 3 | `[41]~[43]` | Decrypt/set return address 0x411919, unchanged via `+[VM_Context+0x2C]` (=0), change ESI with `add esi,eax` |
| Phase 4 | `[44]~[70]` | Restore VM_Context slot values to the stack (trio `0x41f566 → 0x421482 → 0x41fb23` ×9) |
| Phase 5 | `[71]` | `popad`/`popfd` + `ret` → return to 0x411919 |

Going back to the original, it comes down to this one line.

```asm
mov [ebp-8], 0x12345678
; return to 0x411919 → read [ebp-8] and push → call (used as a function argument)
```

## [0x0A] Conclusion

The operation of putting one constant into a local variable and passing it to a function unfolded into a flow of calling 14 handlers 72 times. Because the value, the offset, and the return address are all encrypted in `.VCODE` and unpacked at runtime with a decoder handler, you have to follow the handlers all the way through even to restore a single line. The purpose mentioned in Part 1, "exponentially increasing analysis time," is thus realized at the handler level like this.

That said, the exact purposes of the `+0x20`·`+0x24`·`+0x28`(6)·`+0x2C`·`+0x34` slots, and the meaning of the `+0x2C` that `[42]` adds when it's non-zero — the parts that can't be concluded from a single trace — I've left unresolved. Next time I'll analyze a more complex piece of code with branches, loops, and function calls mixed in, and try to fill in these blanks.
