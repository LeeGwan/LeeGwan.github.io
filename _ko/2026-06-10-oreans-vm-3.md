## [0x00] Overview

[1부](https://leegwan.github.io/ko/posts/2026-05-25-oreans-vm-1/)에서 `.VCPU`/`.VCODE` 구조와 Control Code → Main Handler → Opcode Handler 흐름을, [2부](https://leegwan.github.io/ko/posts/2026-05-30-oreans-vm-2/)에서 14종 핸들러를 하나씩 따라가며 `mov [ebp-8], 0x12345678` 한 줄이 어떻게 처리되는지 끝까지 봤습니다.

이번 글은 직접 만든 Intel Pin 툴(`VMAnalyzer`)의 소스를 **한 줄도 빼놓지 않고** 설명합니다. 어떤 함수를 왜 썼고, 그 인자를 왜 그렇게 넘겼는지까지요. Pin을 처음 보는 분도 따라올 수 있게 풀어 썼습니다.

 **Note:** 따라 하려면 Intel Pin이 필요합니다. 이 툴은 아래 버전 기준으로 작성했습니다.
 - **Download:** [https://www.intel.com/content/www/us/en/developer/articles/tool/pin-a-binary-instrumentation-tool-downloads.html](https://www.intel.com/content/www/us/en/developer/articles/tool/pin-a-binary-instrumentation-tool-downloads.html)
 - **Version:** Pin 3.31 (EOL)
 - **실행:** `pin -logfile pin.log -t [내 Pin DLL 경로] -- [분석할 타겟 경로]`

## [0x01] Intel Pin이 뭐고, 왜 이걸로 만들었나

**[-] Intel Pin = 동적 바이너리 계측(DBI) 도구**

디버거로 분석할 땐 중단점을 걸고 멈춰서 메모리를 들여다봅니다. 그런데 "이 함수가 만진 메모리 주소를 전부 뽑아줘" 같은 건 중단점으로 일일이 하기엔 너무 번거롭습니다. Pin은 이런 걸 자동화해 줍니다. 대상 EXE를 고치지 않고, 실행되는 명령 사이사이에 내 코드로 들어오는 지점을 만들어 그 순간의 레지스터·메모리 값을 읽거나 기록한 뒤 다시 원래 흐름으로 돌아갑니다. 소스도, 재컴파일도 필요 없습니다.

**[-] 왜 이 툴을 만들었나**

실제 가상화된 코드를 손으로 따라가다 보면, 정작 의미 있는 `vm_handler` 말고도 가짜 핸들러·난독화·안티디버깅이 잔뜩 끼어 있습니다. 이 노이즈 때문에 리버싱 속도가 확 느려지고 금방 지칩니다. 그래서 "차라리 한 번 실행시켜서, 이 VM이 대략 어떤 구조로 도는지 — 어떤 핸들러가 무슨 메모리를 만지는지 — 를 로그로 떠두면, 나중에 진짜 분석할 때 어디를 봐야 하는지 지도가 되겠다" 싶었습니다. 이 글의 툴은 거기서 출발했습니다.

**[-] Pin은 콜백으로 동작한다**

Pin은 콜백 방식으로 설계돼 있습니다. `INS_AddInstrumentFunction` 같은 함수로 "명령마다 콜백 걸 기회"를 등록해 두면, Pin이 코드를 JIT 컴파일할 때 그 명령을 한 번 보여줍니다. 주의할 점은, 그 자리에서 바로 분석하는 게 아니라 `INS_InsertCall`로 *실행될 때마다 돌 콜백*을 거는 것입니다. 정리하면 등록 쪽 함수(`IMG_Callbackfunc`, `INS_Callbackfunc`, `AnalyzeInstruction`)는 명령을 훑으며 "무엇을 후킹할지"만 고르고, 거기서 끼운 콜백(`OnHandlerEnter`, `OnMemRead`, `OnMemWriteAfter`)이 실행 때마다 실제 값을 찍습니다.

**[-] IPOINT와 IARG**

끼우는 콜백은 명령 기준 어디에 걸지(`IPOINT_BEFORE` 실행 직전 / `IPOINT_AFTER` 직후 / `IPOINT_TAKEN_BRANCH` 분기가 일어나는 경로)를 정하고, 필요한 값은 `IARG_*` 인자 지정자로 받아옵니다. 예를 들어 `IARG_REG_VALUE, REG_EAX`는 "EAX 값을 넘겨줘", `IARG_MEMORYREAD_EA`는 "이번 읽기의 주소를 넘겨줘"라는 뜻입니다. 인자 목록은 항상 `IARG_END`로 끝냅니다.

## [0x02] VMCONTEXT 구조체 (PIN_CLASS.h)

먼저 헤더입니다. 2부에서 떠본 `VM_Context`(베이스 `0x41E600`)를 그대로 구조체로 옮긴 것입니다.

```cpp
#pragma pack(push, 1)
typedef struct _VMCONTEXT
{
    uint32_t VM_EBP;                // +0x00: 가상 EBP
    uint32_t VM_EDI;                // +0x04: 가상 EDI
    uint32_t VM_EBX;                // +0x08: 가상 EBX
    uint32_t VM_EAX;                // +0x0C: 가상 EAX
    uint32_t VM_ESI;                // +0x10: 가상 ESI
    uint32_t VM_EDX;                // +0x14: 가상 EDX
    uint32_t VM_ECX;                // +0x18: 가상 ECX
    uint32_t VM_EFLAGS;             // +0x1C: 가상 EFLAGS
    uint32_t unk_20;                // +0x20: 미상
    uint32_t unk_24;                // +0x24: 미상
    uint32_t unk_28;                // +0x28: 미상
    uint32_t unk_2C;                // +0x2C: 미상
    uint32_t init_flag;             // +0x30: Control Code lock cmpxchg 대상
    uint32_t unk_34;                // +0x34: 미상
    ADDRINT* handler_table[0x96];   // +0x38: 핸들러 주소 0x96개 배열
} VMCONTEXT, *LPVMCONTEXT;
#pragma pack(pop)
```

한 줄씩 보겠습니다.

- **`VM_EBP` ~ `VM_EFLAGS` (+0x00 ~ +0x1C)** — 2부에서 확인한 8개 가상 레지스터 슬롯입니다. `pushad`/`pushfd`로 백업된 Real CPU 값이 여기로 복사됩니다.
- **`unk_20` ~ `unk_34`** — 용도가 덜 밝혀진 슬롯들입니다. 구조체 오프셋을 끊기지 않게 유지하려고 이름만 붙여 자리를 채웠습니다.
- **`init_flag` (+0x30)** — Control Code가 `lock cmpxchg [edi+30]`으로 건드리는 슬롯입니다. 트레이스 마지막 `[0x47]`에서 이 슬롯을 만지는 게 그대로 보입니다.
- **`handler_table[0x96]` (+0x38)** — Opcode 핸들러 주소 배열입니다. Main Handler가 `jmp [edi+eax*4]`로 이 테이블을 인덱싱해 분기합니다. 개수 `0x96`은 2부에서 확인한 핸들러 테이블 크기입니다.

아래는 실제로 떠본 `0x41E600` 덤프입니다. 위 구조체가 이 메모리에 그대로 겹쳐진다고 보면 됩니다.

[![VM_Context 메모리 덤프](/assets/img/Code Virtualization/Code Virtualization-3/vm_context_dump.png)](/assets/img/Code Virtualization/Code Virtualization-3/vm_context_dump.png)

*0x41E600 — 앞쪽은 레지스터 슬롯(+0x00~+0x1C), +0x38부터는 핸들러 주소 테이블*

`0x41E600`의 첫 줄 `E4 FE 19 00`이 `VM_EBP = 0019FEE4`, `+0x1C`의 `16 02 00 00`이 `VM_EFLAGS = 0x216`, `+0x38`부터의 `0E 0D 42 00 …`이 `handler_table[0] = 0x00420D0E`로 떨어집니다. 이 베이스 주소(`g_vmContextAddr`)를 알아야, 나중에 "이 주소가 `VM_EAX` 슬롯이다" 같은 분류가 가능해집니다.

## [0x03] 진입점과 콜백 등록

`main`은 클래스 객체 하나를 만들어 초기화하고 시작하는 게 전부입니다.

```cpp
int main(int argc, char* argv[])
{
    PIN_CLASS g_bPIN_CLASS;
    if (!g_bPIN_CLASS.pinClassInit(argc, argv))   // 초기화 실패 시 종료
        return -1;
    g_bPIN_CLASS.pinClassStart();                 // 계측 시작 (반환 안 함)
    return 0;
}
```

생성자에서 `PIN_InitSymbols()`를 부릅니다. 심볼 정보(함수 이름 등)를 쓰기 위한 준비인데, Pin 규약상 `PIN_Init` 전에 호출해 둡니다.

```cpp
PIN_CLASS::PIN_CLASS() { PIN_InitSymbols(); }
```

초기화는 `PIN_Init`을 부르는 게 핵심입니다.

```cpp
bool PIN_CLASS::pinClassInit(int argc, char* argv[])
{
    if (PIN_Init(argc, argv) != FALSE) { std::cout << "[PIN] Init failed\n"; return false; }
    std::cout << "[PIN] Init success\n";
    return true;
}
```

- **`PIN_Init(argc, argv)`** — Pin 시스템을 초기화하고 커맨드라인을 파싱합니다. 여기서 헷갈리기 쉬운 게 **반환값의 의미**입니다. `PIN_Init`은 "에러가 있으면 TRUE"를 돌려줍니다. 그래서 `!= FALSE`(에러 있음)일 때 실패 처리하고 `false`를 반환합니다. 직관과 반대라 주의해야 합니다.

이제 콜백을 등록하고 프로그램을 시작합니다.

```cpp
void PIN_CLASS::pinClassStart()
{
    IMG_AddInstrumentFunction(IMG_Callbackfunc, this);   // 이미지 로드 콜백
    INS_AddInstrumentFunction(INS_Callbackfunc, this);   // 명령어 계측 콜백
    PIN_AddFiniFunction(Fini_Callbackfunc, this);        // 종료 콜백
    PIN_StartProgram();                                  // 절대 반환하지 않음
}
```

- **`IMG_AddInstrumentFunction(fun, val)`** — 이미지(EXE/DLL)가 로드될 때마다 `fun`을 부르게 등록합니다. 두 번째 인자 `val`은 콜백 호출 시 그대로 다시 전달됩니다. 여기서 `this`를 넘기는 이유는, 콜백이 `static` 함수라 멤버에 직접 접근할 수 없기 때문입니다. `this`를 받아두면 콜백 안에서 인스턴스로 캐스팅해 멤버 함수를 부를 수 있습니다.
- **`INS_AddInstrumentFunction`** — 명령어 단위 계측 콜백을 등록합니다. (마찬가지로 `this` 전달)
- **`PIN_AddFiniFunction`** — 프로그램 종료 직전에 부를 콜백입니다. 여기서 핸들러 요약을 출력합니다.
- **`PIN_StartProgram()`** — 대상 실행을 시작합니다. 이 함수는 **반환하지 않습니다**(스택까지 정리해 버립니다). 그래서 뒤의 `return 0;`은 사실상 도달하지 않습니다.

## [0x04] 콜백 3종

```cpp
VOID PIN_CLASS::IMG_Callbackfunc(IMG img, VOID* v)
{
    if (!IMG_IsMainExecutable(img)) return;            // 메인 EXE만
    static_cast<PIN_CLASS*>(v)->ProcessImage(img);     // v = 아까 넘긴 this
}
```

- **`IMG_IsMainExecutable(img)`** — 로드되는 이미지가 메인 실행 파일인지 검사합니다. VM 섹션(`.v-lizer`)은 대상 EXE 안에 있으므로, 따라 들어오는 시스템 DLL들은 걸러냅니다.
- **`static_cast<PIN_CLASS*>(v)`** — `pinClassStart`에서 넘긴 `this`를 다시 인스턴스로 복원해, 멤버 함수 `ProcessImage`를 호출합니다.

```cpp
VOID PIN_CLASS::INS_Callbackfunc(INS ins, VOID* v)
{
    ADDRINT ip = INS_Address(ins);
    if (!g_vmStart || !g_vmEnd || ip < g_vmStart || ip > g_vmEnd) return;  // VM 범위 밖이면 무시
    static_cast<PIN_CLASS*>(v)->AnalyzeInstruction(ins);
}
```

- **`INS_Address(ins)`** — 이 명령의 주소(IP)를 얻습니다.
- **범위 게이트** — `g_vmStart`/`g_vmEnd`가 아직 안 잡혔거나(`!g_vmStart`), IP가 VM 섹션 밖이면 바로 빠집니다. 대상 전체가 아니라 **VM 영역 안의 명령만** 계측하겠다는 뜻입니다. 안 그러면 프로그램 전체가 다 후킹돼서 느려지고 로그도 폭발합니다.

```cpp
VOID PIN_CLASS::Fini_Callbackfunc(INT32 code, VOID* v)
{
    DumpHandlerSummary();
    std::cout << "\n[PIN] target exited (code " << code << ")\n";
}
```

종료 시 핸들러 요약표를 찍고 종료 코드를 출력합니다.

## [0x05] VM 영역 탐지 (ProcessImage)

```cpp
void PIN_CLASS::ProcessImage(const IMG& img)
{
    for (SEC sec = IMG_SecHead(img); SEC_Valid(sec); sec = SEC_Next(sec))
        if (SEC_Name(sec) == kVmSection)              // ".v-lizer"
        {
            g_vmStart = SEC_Address(sec);
            g_vmEnd   = g_vmStart + SEC_Size(sec);
            std::cout << "[PIN] " << kVmSection << " @ " << StringFromAddrint(g_vmStart) << "\n";
            break;
        }
}
```

- **`IMG_SecHead` / `SEC_Valid` / `SEC_Next`** — 이미지의 섹션들을 처음부터 끝까지 순회하는 표준 패턴입니다. `SEC_Valid`가 false면 섹션 끝입니다.
- **`SEC_Name(sec) == kVmSection`** — 섹션 이름이 우리가 찾는 VM 섹션(이 샘플은 `.v-lizer`)인지 비교합니다. Oreans 계열 가상화 섹션 이름이 샘플마다 다르므로, 다른 대상이면 이 상수만 바꾸면 됩니다.
- **`SEC_Address` / `SEC_Size`** — 그 섹션의 시작 주소와 크기를 받아 `[g_vmStart, g_vmEnd]` 범위를 확정합니다. 이 범위가 위 `INS_Callbackfunc`의 게이트로 쓰입니다.

## [0x06] 상태 제어 — PUSHAD/POPAD

VM은 Control Code의 `pushad`/`pushfd`로 진입해 `popad`/`popfd`로 빠져나옵니다(1부·2부 참고). 그 경계를 그대로 스위치로 씁니다.

```cpp
VOID PIN_CLASS::TurnOnVmSwitch()
{
    g_vmActive = true;
    // VM 섹션 시작 + 0x8a3 위치에 저장된 vm_context 포인터를 안전하게 읽어옴
    PIN_SafeCopy(&g_vmContextAddr,
                 (VOID*)(g_vmStart + kVmContextPtrOffset),
                 sizeof(g_vmContextAddr));
    g_vmContext = reinterpret_cast<VMCONTEXT*>(g_vmContextAddr);
}
VOID PIN_CLASS::TurnOffVmSwitch() { g_vmActive = false; }
BOOL PIN_CLASS::CheckIfVmActive()  { return g_vmActive ? TRUE : FALSE; }
```

- **`g_vmActive = true`** — 추적 켜기 플래그입니다. 이게 켜져 있는 동안만 `AnalyzeInstruction`이 분석 루틴을 끼웁니다.
- **`PIN_SafeCopy(dst, src, size)`** — `src`의 메모리를 `dst`로 안전하게 복사합니다. 일반 `memcpy`와 달리, 접근 불가능한 주소여도 크래시 없이 복사된 바이트 수만 돌려줍니다. 런타임 메모리를 읽을 땐 항상 이걸 씁니다.
  - 여기서는 `g_vmStart + 0x8a3` 위치에서 4바이트를 읽어 `g_vmContextAddr`에 담습니다. **0x8a3**은 이 샘플에서 VM 섹션 시작으로부터 vm_context(`0x41E600`)가 저장된 자리까지의 오프셋입니다. 이 값을 읽어와야 영역 분류가 가능해집니다. (다른 샘플이면 이 오프셋도 달라집니다.)
- **`reinterpret_cast<VMCONTEXT*>`** — 읽어온 주소를 구조체 포인터로 캐스팅해, 필요하면 슬롯에 이름으로 접근할 수 있게 해둡니다.

`POPAD`(이탈)는 조건부로 끕니다.

```cpp
else if (op == XED_ICLASS_POPAD)
{
    INS_InsertIfCall(ins,   IPOINT_BEFORE, (AFUNPTR)CheckIfVmActive, IARG_END);
    INS_InsertThenCall(ins, IPOINT_BEFORE, (AFUNPTR)TurnOffVmSwitch, IARG_END);
}
```

- **`INS_InsertIfCall` + `INS_InsertThenCall`** — 짝으로 동작합니다. "If" 루틴(`CheckIfVmActive`)이 0이 아닌 값을 돌려줄 때만, 바로 뒤 "Then" 루틴(`TurnOffVmSwitch`)이 실행됩니다. 즉 **VM이 활성일 때의 `popad`만** 이탈로 처리합니다. 쉽게 말하면 pushad가 가상화(또는 VM 처리)의 시작 지점이라면, popad는 가상화 처리가 끝났으므로 저장해 두었던 실제 CPU 레지스터 값을 복구하는 시점이라고 볼 수 있습니다.

## [0x07] 명령어 계측 (AnalyzeInstruction)

VM이 활성인 동안, VM 범위 안 명령마다 어떤 분석 루틴을 끼울지 결정하는 본체입니다. 먼저 진입/이탈 스위치부터 답니다.

```cpp
void PIN_CLASS::AnalyzeInstruction(const INS& ins)
{
    OPCODE op = INS_Opcode(ins);

    if (op == XED_ICLASS_PUSHAD)
        INS_InsertCall(ins, IPOINT_BEFORE, (AFUNPTR)TurnOnVmSwitch, IARG_END);
    else if (op == XED_ICLASS_POPAD)
    {
        INS_InsertIfCall(ins,   IPOINT_BEFORE, (AFUNPTR)CheckIfVmActive, IARG_END);
        INS_InsertThenCall(ins, IPOINT_BEFORE, (AFUNPTR)TurnOffVmSwitch, IARG_END);
    }
    ...
```

- **`INS_Opcode(ins)`** — 명령의 종류(opcode)를 얻습니다. `XED_ICLASS_PUSHAD`/`POPAD`는 Pin이 디코딩한 `pushad`/`popad`를 가리킵니다.
- **`PUSHAD` → `IPOINT_BEFORE`에 `TurnOnVmSwitch`** — `pushad`가 *실행되기 직전*에 스위치를 켭니다. 이 시점이면 vm_context 포인터가 이미 자리잡아 있어 안전하게 읽을 수 있습니다.
- **`POPAD` → `InsertIfCall(CheckIfVmActive)` + `InsertThenCall(TurnOffVmSwitch)`** — `popad`는 VM이 레지스터를 복구하고 빠져나갈 때 나옵니다. 그래서 *실행되기 직전*에 스위치를 끕니다. 다만 무조건 끄지 않고 "If/Then" 짝으로 겁니다. 먼저 `CheckIfVmActive`가 현재 VM이 켜져 있는지 확인해서 0이 아닌 값을 돌려줄 때만, 바로 뒤 `TurnOffVmSwitch`가 실행돼 스위치를 끕니다. 프로그램에는 VM과 무관한 `popad`도 있을 수 있는데, 그런 경우는 VM이 꺼져 있으니 `CheckIfVmActive`가 막아 줍니다.

### [-] 핸들러 진입 — 디스패처 점프 후킹

2부의 Main Handler는 `jmp [edi+eax*4]`로 핸들러 테이블을 인덱싱해 분기합니다. 그 패턴을 정확히 짚습니다.

```cpp
if (INS_IsIndirectControlFlow(ins) && !INS_IsRet(ins) &&
    INS_IsValidForIpointTakenBranch(ins) &&
    INS_MemoryBaseReg(ins)  == REG_EDI &&
    INS_MemoryIndexReg(ins) == REG_EAX &&
    INS_MemoryScale(ins)    == 4)
{
    INS_InsertCall(ins, IPOINT_TAKEN_BRANCH, (AFUNPTR)OnHandlerEnter,
        IARG_REG_VALUE, REG_EAX,        // 디코딩된 opcode(테이블 인덱스)
        IARG_BRANCH_TARGET_ADDR,        // 점프해 들어갈 실제 핸들러 주소
        IARG_END);
}
```

조건 하나하나가 `jmp [edi+eax*4]`를 정확히 집어내기 위한 필터입니다.

- **`INS_IsIndirectControlFlow`** — 간접 제어 흐름인가. 점프든 콜이든 `ret`이든, 분기 **대상 주소가 레지스터나 메모리에서 오는** 제어 흐름 명령이면 참입니다. `jmp [edi+eax*4]`처럼 대상이 즉시값/IP 상대 오프셋이 아니라 메모리에서 읽혀 오는 경우가 여기 해당합니다.
- **`!INS_IsRet`** — `ret`은 대상을 스택(메모리)에서 가져오므로 위 조건에 함께 걸립니다. 우리가 찾는 건 디스패처 점프지 함수 복귀가 아니므로 `ret`은 빼 줍니다.
- **`INS_IsValidForIpointTakenBranch`** — 이 명령에 `IPOINT_TAKEN_BRANCH` 위치로 분석 루틴을 끼울 수 있는지 미리 묻는 함수입니다. `IPOINT_TAKEN_BRANCH`는 "분기가 실제로 일어나는 경로", 즉 제어가 대상 주소로 넘어가는 바로 그 순간에만 콜을 거는 지점인데, 이건 분기 계열 명령에만 의미가 있습니다. 분기가 아닌 명령에 `IPOINT_TAKEN_BRANCH`를 걸면 Pin이 에러를 냅니다. 그래서 `INS_InsertCall`을 호출하기 전에 이 함수로 참인지 확인해, 우리가 노리는 디스패처 점프에만 안전하게 후킹하도록 막아 둡니다.
- **`MemoryBaseReg == REG_EDI` / `MemoryIndexReg == REG_EAX` / `MemoryScale == 4`** — 메모리 오퍼랜드가 정확히 `[EDI + EAX*4]` 형태인지 검사합니다. base=EDI, index=EAX, scale=4 — 곧 핸들러 테이블 인덱싱 패턴입니다. 이 세 조건으로 다른 간접 점프와 구분합니다.

후킹은 `IPOINT_TAKEN_BRANCH`(분기가 실제로 일어나는 경로)에 겁니다. 그래야 핸들러로 넘어가는 바로 그 순간을 잡습니다. 인자는:

- **`IARG_REG_VALUE, REG_EAX`** — 그 시점의 EAX 값. Main Handler가 디코딩을 끝낸 opcode(테이블 인덱스)가 EAX에 들어 있습니다.
- **`IARG_BRANCH_TARGET_ADDR`** — 이 분기가 점프할 대상 주소. 곧 실제 핸들러 주소입니다. `[edi+eax*4]`를 우리가 직접 계산하지 않아도 Pin이 풀어서 줍니다.

### [-] LEA / 읽기 / 쓰기

```cpp
if (INS_IsLea(ins)) {
    REG  baseReg    = INS_MemoryBaseReg(ins);
    BOOL isStackLea = (baseReg == REG_ESP || baseReg == REG_EBP);
    INS_InsertPredicatedCall(ins, IPOINT_BEFORE, (AFUNPTR)OnLea,
        IARG_EXPLICIT_MEMORY_EA,                 // lea가 계산한 주소
        IARG_REG_VALUE, REG_ESP, IARG_BOOL, isStackLea, IARG_END);
}
```

- **`INS_IsLea`** — `lea` 명령인가. `lea`는 주소를 계산만 하고 메모리는 읽지 않습니다. 그래서 메모리 읽기/쓰기 후킹과 따로 잡아야 합니다.
- **`IARG_EXPLICIT_MEMORY_EA`** — `lea`가 계산한 유효 주소를 받습니다. Pin 문서가 `lea` 계측용으로 명시한 인자라, 존재하지도 않는 "메모리 접근" 없이 계산된 주소만 깔끔하게 가져옵니다.
- **`isStackLea`** — base가 ESP/EBP면 스택 주소 계산으로 보고 플래그로 넘깁니다(로그 라벨링용).
- **`INS_InsertPredicatedCall`** — `INS_InsertCall`과 거의 같지만, predicate가 붙는 명령(CMOVcc, REP 등)에서 조건이 거짓이면 분석 루틴을 건너뜁니다. 실제로 실행된 동작만 로그에 남기려고 predicated 버전을 씁니다.

```cpp
if (INS_IsMemoryRead(ins)) {
    BOOL isStackRead = INS_IsStackRead(ins);
    INS_InsertPredicatedCall(ins, IPOINT_BEFORE, (AFUNPTR)OnMemRead,
        IARG_MEMORYREAD_EA, IARG_MEMORYREAD_SIZE,   // 읽기 주소 + 크기
        IARG_REG_VALUE, REG_ESP, IARG_BOOL, isStackRead, IARG_END);
}
```

- **`INS_IsMemoryRead`** — 메모리를 읽는 명령인가.
- **`IARG_MEMORYREAD_EA` / `IARG_MEMORYREAD_SIZE`** — 이번 읽기의 유효 주소와 바이트 크기입니다. 이 중 `IARG_MEMORYREAD_EA`는 `INS_IsMemoryRead`가 참인 명령에서, 그리고 `IPOINT_BEFORE`에서만 유효합니다. 여기서는 BEFORE에 걸므로 문제없습니다.
- **`INS_IsStackRead`** — 이 읽기가 스택 접근인지 Pin이 판정해 줍니다. 내부적으로는 ESP/RSP 또는 EBP/RBP 기준 접근인지를 봅니다. 다만 프레임 포인터 없이 컴파일돼 EBP/RBP가 범용 레지스터로 쓰이는 코드에서는 잘못 판정될 수 있다는 점은 알아둬야 합니다. 결과는 라벨링에 씁니다.

```cpp
if (INS_IsMemoryWrite(ins)) {
    BOOL isStackWrite = INS_IsStackWrite(ins);
    INS_InsertPredicatedCall(ins, IPOINT_BEFORE, (AFUNPTR)OnMemWriteBefore,
        IARG_MEMORYWRITE_EA, IARG_MEMORYWRITE_SIZE,  // 쓰기 주소 + 크기 (값은 아직 없음)
        IARG_REG_VALUE, REG_ESP, IARG_END);

    if (INS_IsValidForIpointAfter(ins))
        INS_InsertPredicatedCall(ins, IPOINT_AFTER,
            (AFUNPTR)OnMemWriteAfter, IARG_BOOL, isStackWrite, IARG_END);
}
```

쓰기를 BEFORE/AFTER로 쪼갠 이유가 핵심입니다.

- **`IARG_MEMORYWRITE_EA`는 `IPOINT_BEFORE`에서만 유효**합니다. 그런데 그 시점엔 값이 아직 메모리에 안 써졌습니다. 그래서 BEFORE(`OnMemWriteBefore`)에서는 **주소·크기·ESP만** 저장해 둡니다.
- 실제 써진 값은 AFTER(`OnMemWriteAfter`)에서 그 주소를 다시 읽어 확인합니다. `INS_IsValidForIpointAfter`로 이 명령이 IPOINT_AFTER를 지원할 때만 후킹합니다(분기·`call` 등은 fall-through가 없어 안 됨).

## [0x08] 분석 루틴과 영역 분류

### [-] 핸들러 진입 로그 (OnHandlerEnter)

```cpp
VOID PIN_CLASS::OnHandlerEnter(ADDRINT opcode, ADDRINT target)
{
    g_curHandler = target;
    g_curOpcode  = opcode;
    g_inDispatch = false;              // 핸들러 본문 시작 → 필터링 off (여기부터가 원하는 로그)

    HandlerInfo& h = g_handlerStats[target];
    h.callCount++;
    if (h.sampleOpcode.empty()) h.sampleOpcode = StringFromAddrint(opcode);

    LOG("[" + StringFromAddrint(g_step++) + "] op(DECODING)=" +
        StringFromAddrint(opcode) + " handler=" + StringFromAddrint(target) + "\n");
}
```

- **`opcode`/`target`** — 위에서 넘긴 `IARG_REG_VALUE REG_EAX`와 `IARG_BRANCH_TARGET_ADDR`가 그대로 들어옵니다.
- **`g_curHandler`/`g_curOpcode` 갱신** — 지금부터 발생하는 읽기/쓰기가 "어느 핸들러 소속인지" 표시하는 현재 상태 값입니다.
- **`g_inDispatch = false`** — 핸들러 본문이 시작됐으니 필터링을 끕니다. 여기서부터 찍히는 게 우리가 보고 싶은 로그입니다. 반대로 디코더가 만드는 쓰레기 값이 감지되면 `g_inDispatch = true`로 필터링을 켜서 그 구간은 로그에 저장하지 않습니다(아래 [0x09]에서 설명).
- **`g_handlerStats[target]`** — 핸들러 주소를 키로 통계 누적. `callCount++`로 호출 횟수를, 첫 등장 때 `sampleOpcode`에 opcode를 기록합니다.
- **`LOG(...)`** — 트레이스의 `[0x..] op(DECODING)=.. handler=..` 헤더 줄이 여기서 나옵니다. `g_step++`로 스텝 번호를 매깁니다.

### [-] 영역 분류 (RegionName)

읽고 쓰는 주소가 vm_context의 어느 슬롯인지, 스택인지, VCODE인지 분류해 라벨을 붙입니다. 이게 로그를 읽을 수 있게 만드는 핵심입니다.

```cpp
static std::string RegionName(ADDRINT x, ADDRINT esp, BOOL isStack)
{
    if (g_vmContextAddr && x >= g_vmContextAddr &&
        x < g_vmContextAddr + sizeof(VMCONTEXT))     // vm_context 범위 안인가
    {
        UINT32 off = (UINT32)(x - g_vmContextAddr);  // 베이스로부터의 오프셋
        switch (off) {
        case 0x00: return "VM_EBP";  case 0x04: return "VM_EDI";
        case 0x08: return "VM_EBX";  case 0x0C: return "VM_EAX";
        case 0x10: return "VM_ESI";  case 0x14: return "VM_EDX";
        case 0x18: return "VM_ECX";  case 0x1C: return "VM_EFLAGS";
        case 0x30: return "init_flag";
        default:
            if (off >= 0x38) return "handler_table";
            return "VMCTX+" + StringFromAddrint(off);
        }
    }
    if (isStack) return "STACK";
    if (g_vmStart && x >= g_vmStart && x < g_vmEnd) return "Read OPCODE";
    return "";
}
```

- **vm_context 범위 판정** — 주소 `x`가 `[g_vmContextAddr, +sizeof(VMCONTEXT))` 안이면 vm_context 슬롯입니다.
- **`off`로 슬롯 이름 매핑** — 베이스에서 뺀 오프셋으로 `VM_EAX`, `VM_EFLAGS`, `init_flag` 등을 돌려줍니다. `0x38` 이상이면 `handler_table`, 그 외 이름 없는 슬롯은 `VMCTX+오프셋`으로 찍습니다.
- **`STACK`** — vm_context가 아니고 스택 접근이면 `STACK`.
- **`Read OPCODE`** — VM 섹션(`[g_vmStart, g_vmEnd)`) 안의 읽기면 바이트코드 페치로 보고 `Read OPCODE`.
- **그 외** — 빈 문자열을 돌려주고, 호출부에서 `MEM`으로 채웁니다(힙/타깃 데이터).

스택(특히 ESP 기준 접근)을 따로 추적하는 데는 이유가 있습니다. `vm_handler`는 연산을 처리하고 나서 결과를 두 가지 방식으로 다룹니다. 하나는 vm_context의 특정 슬롯에 값을 써넣어 그 자리에서 연산을 마무리하는 경우이고, 다른 하나는 결과나 중간값을 **스택에 쌓아(특히 ESP에 올려)** 다음 핸들러에 넘기는 경우입니다. 후자에서는 한 핸들러가 ESP 쪽에 올려둔 값을 다음 핸들러가 다시 꺼내 처리하면서, 핸들러들이 스택을 매개로 데이터를 주고받습니다. 그래서 이 스택 슬롯의 쓰기/읽기를 `STACK`으로 라벨링해 두면 "어느 핸들러가 무엇을 올려서 어느 핸들러가 받아 가는지"라는 핸들러 간 데이터 흐름이 로그에 드러납니다.

이 분류를 두 함수가 감쌉니다.

```cpp
static std::string Loc(ADDRINT addr, ADDRINT esp, BOOL isStack) {
    std::string r = RegionName(addr, esp, isStack);
    if (r.empty()) r = "MEM";
    return "[" + r + "]" + StringFromAddrint(addr);   // 예: [VM_EAX]0x0041e60c
}
static std::string ValTag(ADDRINT val, ADDRINT esp, BOOL isStack) {
    std::string r = RegionName(val, esp, isStack);
    return r.empty() ? "" : " (->" + r + ")";          // 예: (->STACK)
}
```

- **`Loc`** — 주소를 `[영역]0xADDR` 형태로 라벨링합니다. 분류 실패 시 `MEM`.
- **`ValTag`** — *값* 자체가 또 다른 영역을 가리키면 `(->STACK)`, `(->VM_EAX)` 태그를 붙입니다. 슬롯 주소가 스택을 거쳐 흘러가는 포인터 체인을 따라가기 쉽게 해줍니다.

### [-] 읽기/쓰기 로그

```cpp
VOID PIN_CLASS::OnMemRead(ADDRINT addr, UINT32 size, ADDRINT esp, BOOL isStack)
{
    if (!g_curHandler) return;                    // 핸들러 진입 전이면 무시
    g_handlerStats[g_curHandler].readCount++;     // 통계 누적
    if (g_inDispatch) return;                     // 디코딩 노이즈면 출력만 스킵

    ADDRINT val = 0;
    PIN_SafeCopy(&val, (VOID*)addr, (size < sizeof(ADDRINT)) ? size : sizeof(ADDRINT));
    LOG("      READ  " + Loc(addr, esp, isStack) +
        " val:" + StringFromAddrint(val) + ValTag(val, esp, isStack) + "\n");
}
```

- **`if (!g_curHandler) return`** — 아직 핸들러에 진입하지 않은 상태(디스패처 도입부 등)면 무시합니다.
- **`readCount++`** — 현재 핸들러의 읽기 횟수를 누적합니다.
- **`PIN_SafeCopy`로 값 읽기** — 주소 `addr`에서 값을 안전하게 읽습니다. 크기는 `(size < 4) ? size : 4`로 **클램프**합니다. 1·2바이트 접근일 때 4바이트를 통째로 읽어 옆 메모리까지 끌고 오는 걸 막으려는 것입니다.
- **`LOG`** — `Loc`로 주소를, `ValTag`로 값이 가리키는 영역을 붙여 `READ [영역]주소 val:값 (->영역)` 한 줄을 찍습니다.

쓰기는 BEFORE에서 좌표만 저장하고:

```cpp
VOID PIN_CLASS::OnMemWriteBefore(ADDRINT addr, UINT32 size, ADDRINT esp)
{
    g_wAddr = addr; g_wSize = size; g_wEsp = esp;   // 값은 아직 안 써짐 → 좌표만 보관
}
```

AFTER에서 실제 써진 값을 읽어 찍습니다(노이즈 판정 포함, 다음 절).

## [0x09] 노이즈 필터링 (OnMemWriteAfter)

처음엔 메모리 접근을 전부 찍었더니 Main Handler의 fetch/decode 루틴이 만들어내는 쓰레기 값이 로그를 도배했습니다. 그래서 디코드 루틴이 스택에 쓰는 **시그니처 값**을 감지해, 다음 핸들러 진입 전까지 출력을 건너뜁니다.

```cpp
static const ADDRINT kDispatchMagic1 = 0x4b82;
static const ADDRINT kDispatchMagic2 = 0x25c5;

VOID PIN_CLASS::OnMemWriteAfter(BOOL isStack)
{
    if (!g_curHandler) return;
    g_handlerStats[g_curHandler].writeCount++;     // 통계 누적

    ADDRINT val = 0;
    PIN_SafeCopy(&val, (VOID*)g_wAddr, (g_wSize < sizeof(ADDRINT)) ? g_wSize : sizeof(ADDRINT));

    // 써진 값이 디코더 시그니처면, 다음 핸들러 진입까지를 노이즈로 간주
    if (val == kDispatchMagic1 || val == kDispatchMagic2)
        g_inDispatch = true;
    if (g_inDispatch) return;                      // 출력만 스킵

    LOG("      WRITE " + Loc(g_wAddr, g_wEsp, isStack) +
        " <- " + StringFromAddrint(val) + ValTag(val, g_wEsp, isStack) + "\n");
}
```

- **BEFORE에서 저장해둔 좌표(`g_wAddr` 등)로 값 읽기** — 이제 쓰기가 끝났으니, 그 주소를 다시 읽으면 *실제 저장된 값*이 나옵니다. 로그에 찍히는 값이 저장의 진짜 결과인 이유입니다.
- **시그니처 감지** — 써진 값이 `0x4b82`/`0x25c5`면 디코더가 동작을 시작한 것으로 보고 `g_inDispatch = true`로 필터링을 켭니다. 이 구간의 쓰레기 값은 로그에 저장하지 않습니다.
- **`g_inDispatch`면 출력 스킵** — 이 플래그는 다음 `OnHandlerEnter`에서 다시 `false`로 풀립니다. 즉 디코드 노이즈는 묻히고, 핸들러 본문의 의미 있는 접근만 트레이스에 남습니다.

 **Note:** 카운터 증가(`readCount++`/`writeCount++`)가 `g_inDispatch` 체크보다 **앞**에 있습니다. 그래서 종료 시 요약표의 reads/writes는 노이즈까지 포함한 raw 카운트이고, 트레이스에 실제로 찍힌 줄 수보다 클 수 있습니다.

마지막으로 요약 출력입니다.

```cpp
VOID PIN_CLASS::DumpHandlerSummary()
{
    LOG("\n================= HANDLER SUMMARY =================\n");
    LOG("handler        callscount       readscount     writescount    opcode(Decoding)\n");
    for (auto& kv : g_handlerStats) {
        const HandlerInfo& h = kv.second;
        LOG(StringFromAddrint(kv.first) + "   " +
            StringFromAddrint(h.callCount) + "    " + StringFromAddrint(h.readCount) + "    " +
            StringFromAddrint(h.writeCount) + "    " + h.sampleOpcode + "\n");
    }
    LOG("==================================================\n");
}
```

누적해 둔 통계를 그대로 출력하는 부분입니다. 각 행이 핸들러별 호출/읽기/쓰기 횟수를 보여줍니다(`std::map`이라 핸들러 주소 오름차순으로 정렬돼 나옵니다).

## [0x0B] Conclusion

손으로 72스텝을 따라가던 일을, 이제는 실행 한 번으로 "어떤 핸들러가, 몇 번, 무엇을" 찍어주는 데까지 줄였습니다. Pin의 명령어 계측 위에 디스패처 후킹(`jmp [edi+eax*4]` + `IPOINT_TAKEN_BRANCH`), 영역 분류(`VM_Context`/`STACK`/`Read OPCODE`/`MEM`), 노이즈 필터링(시그니처 `0x4b82`/`0x25c5`) 세 가지만 얹으면, 가비지에 파묻힌 트레이스를 읽을 수 있는 형태로 끌어낼 수 있었습니다.

물론 이 툴은 트레이스를 깔끔하게 뽑아줄 뿐, 각 핸들러의 의미까지 알려주진 않습니다. 거기서부터는 2부처럼 직접 따라가며 복원해야 합니다.
