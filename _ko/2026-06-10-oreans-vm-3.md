## [0x00] Overview

[2부](https://leegwan.github.io/ko/posts/2026-05-30-oreans-vm-2/)에서는 단 한 줄의 `mov [ebp-8], 0x12345678` 원본 코드가 가상화 블록 내부에서 어떻게 14개의 핸들러와 72번의 호출로 팽창하는지 디버거로 한 땀 한 땀 추적했습니다.

하지만 매번 이렇게 수동으로 모든 핸들러 트레이스를 따라갈 수는 없습니다. 그래서 Control Code 진입점(`PUSHAD`)부터 탈출점(`POPAD`)까지의 모든 핸들러 분기(Dispatcher)와 메모리 R/W 내역을 실시간으로 덤프하는 **자동화 분석 툴(PinVMTracer)**을 Intel Pin API를 이용해 제작했습니다.

이번 포스팅에서는 이 툴을 개발하면서 전역 변수를 활용한 상태 관리 설계 변수들의 의미, 영역 필터링 메커니즘, 그리고 각 분석 함수와 계측 API인 인자값들의 기술적 역할을 상세히 분석해 보겠습니다.

## [0x01] The VM_Context Structure

코드 분석에 앞서 툴이 실시간으로 참조하게 될 핵심 데이터 구조인 `VM_Context` 구조체 정의입니다. 가상화 엔진은 Real CPU의 레지스터를 백업하고 자체적인 연산을 수행하기 위해 메모리 상에 VCPU의 상태를 저장하는 컨텍스트를 유지합니다.

![VM_Context Dump](/assets/img/Code Virtualization/Code Virtualization-1/vm_context_after.png)

위 덤프 이미지와 같이 메모리에 실시간으로 기록되는 상태를 분석 툴에서 매핑하기 위해 다음과 같은 `_VMCONTEXT` 구조체를 정의했습니다.

```cpp
#pragma pack(push, 1)
typedef struct _VMCONTEXT
{
    uint32_t VM_EBP;                // +0x00: Virtual EBP register
    uint32_t VM_EDI;                // +0x04: Virtual EDI register
    uint32_t VM_EBX;                // +0x08: Virtual EBX register
    uint32_t VM_EAX;                // +0x0C: Virtual EAX register
    uint32_t VM_ESI;                // +0x10: Virtual ESI register
    uint32_t VM_EDX;                // +0x14: Virtual EDX register
    uint32_t VM_ECX;                // +0x18: Virtual ECX register
    uint32_t VM_EFLAGS;             // +0x1C: Virtual EFLAGS register
    uint32_t unk_20;                // +0x20: Unknown field
    uint32_t unk_24;                // +0x24: Unknown field
    uint32_t unk_28;                // +0x28: Unknown field
    uint32_t unk_2C;                // +0x2C: Unknown field
    uint32_t init_flag;             // +0x30: Control Code lock cmpxchg target
    uint32_t unk_34;                // +0x34: Unknown field
    ADDRINT* handler_table[0x96];   // +0x38: Array of 0x96 virtual handler addresses
} VMCONTEXT, * LPVMCONTEXT;
#pragma pack(pop)

### JIT vs Runtime 동기화 조율 변수
g_vmActive: 현재 실행 흐름이 VM 블록 내부에 머물고 있는지 여부를 판별하는 플래그입니다. PUSHAD 감시 시 활성화되고 POPAD를 감시할 때 비활성화됩니다.

g_vmStart, g_vmEnd: 타겟 모듈의 .v-lizer 가상화 섹션 범위의 경계 주소를 보관합니다. 계측(Instrumentation) 오버헤드를 최소화하기 위해 주소 필터링용으로 활용됩니다.

g_vmContextAddr, g_vmContext: 타겟 프로세스 내부에서 실시간으로 갱신되는 가상 레지스터 구조체의 기본 메모리 주소 및 데이터를 안전하게 덤프해오기 위한 백업 포인터입니다.

std::unique_ptr이나 클래스 자체를 전역 인스턴스로 두는 방법도 있으나, JIT 엔진 컴파일러와 런타임 분석 콜백 간의 생명주기 꼬임을 원천 차단하고 메모리 고정성을 확보하기 위해 개별 변수들을 글로벌 데이터 영역(.bss/.data)에 명시적으로 고정시켰습니다.

### 명령어 분석 및 추적 변수
g_curHandler, g_curOpcode, g_step: 현재 실행 중인 가상 핸들러의 주소와 디코딩된 내부 Opcode, 그리고 전체 추적 단계를 기록하기 위한 상태 카운터입니다.

g_wAddr, g_wSize, g_wEsp: 메모리 쓰기 연산을 분석할 때 사용되는 변수입니다. 핀 엔진 구조상 값이 실제로 써지기 전(IPOINT_BEFORE)에 접근 대상 주소와 속성을 미리 보관해 두어야, 연산이 끝난 직후(IPOINT_AFTER)에 메모리에서 변경된 최종 값을 추적할 수 있기 때문에 존재합니다.

g_inDispatch: 가상화 명령어 연산이 끝나고 다음 명령어 핸들러 주소를 계산해 내는 디스패처(Dispatcher) 구간의 연산 노이즈를 식별하고 걸러내기 위한 상태 제어 변수입니다.

```cpp
/// 핸들러 종류별 누적 통계
struct HandlerInfo
{
    UINT32      callCount = 0;
    UINT32      readCount = 0;
    UINT32      writeCount = 0;
    std::string sampleOpcode;
};
std::map<ADDRINT, HandlerInfo> g_handlerStats;

static const std::string kVmSection = ".v-lizer";
static const ADDRINT     kVmContextPtrOffset = 0x8a3;

// 디스패처 복호화 루틴의 시그니처 상수 (이 값을 STACK에 쓰면 디스패처 시작)
static const ADDRINT     kDispatchMagic1 = 0x4b82;
static const ADDRINT     kDispatchMagic2 = 0x25c5;
struct HandlerInfo & g_handlerStats: 프로그램이 끝날 때 가상 핸들러별로 총 몇 번 호출되었고, 메모리를 얼마나 읽고 썼는지 그룹화하여 최종 요약 레포트를 생성하기 위한 보관용 컨테이너입니다.

```
kVmSection, kVmContextPtrOffset: 타겟 분석 대상 가상화 섹션명과 내부 컨텍스트 포인터 주소가 들어있는 오프셋(0x8a3) 상수입니다.

kDispatchMagic1, kDispatchMagic2: 디스패처 복호화 루틴의 시그니처 상수입니다. 디스패처 루틴이 다음 핸들러 주소를 계산하기 위해 스택에 이 고유 상수를 라이트하는 패턴이 감지되면, 흐름이 디스패처 연산 구간으로 전이되었음을 즉시 간파하고 이후 발생하는 연산들을 노이즈로 처리하여 로그 필터링(g_inDispatch = true)을 수행하는 지표가 됩니다.

## [0x03] ProcessImage와 INS_Callbackfunc를 통한 영역 필터링
가상화 엔진 추적 시 핀툴이 윈도우 커널 DLL이나 일반 시스템 영역의 수많은 명령어까지 전부 분석하게 되면 오버헤드로 인해 프로세스가 멈추게 됩니다. 이 코드는 ProcessImage와 주소 필터링 구문을 통해 감시 대상을 엄격하게 제한합니다.

```cpp
void PIN_CLASS::ProcessImage(const IMG& img)
{
    for (SEC sec = IMG_SecHead(img); SEC_Valid(sec); sec = SEC_Next(sec))
        if (SEC_Name(sec) == kVmSection)
        {
            // 타겟으로 하는 주소 영역 정보를 가져옴
            g_vmStart = SEC_Address(sec); 
            g_vmEnd = g_vmStart + SEC_Size(sec);
            std::cout << "[PIN] " << kVmSection << " @ " << StringFromAddrint(g_vmStart) << "\n";
            break;
        }
}
```
ProcessImage: 타겟 바이너리 모듈이 메모리에 로드되는 최초 시점에 실행됩니다. 이 함수의 목적은 내부 데이터를 파싱하는 것이 아니라, 우리가 추적하고자 하는 가상화 영역인 .v-lizer 섹션의 시작 주소(g_vmStart)와 끝 주소(g_vmEnd)의 경계 주소 영역 정보를 정확히 획득하는 것입니다.

```cpp
VOID PIN_CLASS::INS_Callbackfunc(INS ins, VOID* v)
{
    ADDRINT ip = INS_Address(ins);
    
    // 가져온 주소 영역 정보로 필터링 수행
    if (!g_vmStart || !g_vmEnd || ip < g_vmStart || ip > g_vmEnd) return;
    
    static_cast<PIN_CLASS*>(v)->AnalyzeInstruction(ins);
}
INS_Callbackfunc: CPU가 명령어를 JIT 컴파일할 때 명령어 하나하나마다 호출되는 콜백입니다. ProcessImage에서 획득한 g_vmStart와 g_vmEnd 주소 영역 정보를 사용하여, 현재 명령어 주소(ip)가 가상화 영역 내부가 아니라면 묻지도 따지지도 않고 바로 쳐내도록 필터링을 수행합니다. 이 필터링 덕분에 불필요한 시스템 분석 노이즈가 완벽하게 차단됩니다.

[0x04] Control Code 진입과 TurnOnVmSwitch
```
가상화 구조체 데이터를 파싱하여 가져오는 작업은 실행 타이밍이 매우 정교해야 합니다.

```cpp
VOID PIN_CLASS::TurnOnVmSwitch()
{
    g_vmActive = true;
    
    // Control Code가 시작되어 리얼 CPU 레지스터를 저장한 이후 주소를 안전하게 복사
    PIN_SafeCopy(&g_vmContextAddr, (VOID*)(g_vmStart + kVmContextPtrOffset), sizeof(g_vmContextAddr));
    g_vmContext = reinterpret_cast<VMCONTEXT*>(g_vmContextAddr);
}
```
설계적 의도: 가상화 모듈 로드 시점에는 데이터가 유효하지 않습니다. TurnOnVmSwitch()가 호출된 것 자체가 Control Code의 시작을 의미합니다. Control Code가 구동되면서 가상 CPU(VCPU)로 제어권을 넘기기 전에, Real CPU 레지스터와 플래그들을 백업(PUSHAD)하는 시점입니다.

따라서 이 시점에 맞춰 TurnOnVmSwitch 분석 함수가 작동하도록 타이밍을 맞추어, 메모리가 완벽하게 동기화된 상태에서 PIN_SafeCopy로 진짜 컨텍스트 포인터를 가져오고 _VMCONTEXT 구조체 버퍼 주소를 안전하게 매핑하게 됩니다.

## [0x05] 분석 루틴(Analysis Routines) 상세 해설
타겟 프로그램 명령어의 런타임 이벤트를 직접 처리하고 데이터 이름을 판별하는 분석 루틴 코드들의 세부 메커니즘입니다.

```cpp
static std::string RegionName(ADDRINT x, ADDRINT esp, BOOL isStack)
{
    if (g_vmContextAddr && x >= g_vmContextAddr &&
        x < g_vmContextAddr + sizeof(VMCONTEXT))
    {
        UINT32 off = (UINT32)(x - g_vmContextAddr);
        switch (off) {
        case 0x00: return "VM_EBP";  case 0x04: return "VM_EDI";
        case 0x08: return "VM_EBX";  case 0x0C: return "VM_EAX";
        case 0x10: return "VM_ESI";  case 0x14: return "VM_EDX";
        case 0x18: return "VM_ECX";  case 0x1C: return "VM_EFLAGS";
        case 0x30: return "init_flag";
        default:
            if (off >= 0x38) return "handler_table";
            return "VMCTX+" + StringFromAddrint(off);
        }
    }
    
    if (isStack) return "STACK";
    if (g_vmStart && x >= g_vmStart && x < g_vmEnd) return "Read OPCODE";
    return "";
}
```
RegionName: 연산 대상 메모리 주소 x가 들어왔을 때, 해당 주소의 실제 정체를 파악하는 라벨러 함수입니다. 주소가 g_vmContextAddr 범위 내부라면 각 DWORD 오프셋을 역산하여 VM_EBP, VM_EFLAGS 등의 이름을 반환합니다. 만약 스택 플래그(isStack)가 참이면 "STACK"을, 주소가 가상화 섹션 범위 내라면 가상 Opcode 패치 구간인 "Read OPCODE" 라벨을 반환합니다.

```cpp
static std::string Loc(ADDRINT addr, ADDRINT esp, BOOL isStack)
{
    std::string r = RegionName(addr, esp, isStack);
    if (r.empty()) r = "MEM"; 
    return "[" + r + "]" + StringFromAddrint(addr);
}

static std::string ValTag(ADDRINT val, ADDRINT esp, BOOL isStack)
{
    std::string r = RegionName(val, esp, isStack);
    return r.empty() ? "" : " (->" + r + ")";
}
```
Loc: RegionName으로 획득한 명칭 뒤에 헥스 주소 문자열을 결합하여 가시성을 확보해 주는 포맷터입니다. 매핑에 실패하면 일반 힙이나 데이터 주소인 [MEM] 꼬리표를 붙입니다.

ValTag: 메모리에서 읽어 들인 값인 val 자체의 주소가 무언가를 가리키고 있을 때 포인터 추적 관계(예: (->VM_EFLAGS))를 직관적으로 덧붙여 출력해 줍니다.

```cpp
VOID PIN_CLASS::OnHandlerEnter(ADDRINT opcode, ADDRINT target)
{
    g_curHandler = target;
    g_curOpcode = opcode;
    g_inDispatch = false;   // 핸들러 본문 시작 → 노이즈 억제 해제

    HandlerInfo& h = g_handlerStats[target];
    h.callCount++;
    if (h.sampleOpcode.empty()) h.sampleOpcode = StringFromAddrint(opcode);

    LOG("[" + StringFromAddrint(g_step++) + "] op(DECODING)=" + StringFromAddrint(opcode) +
        " handler=" + StringFromAddrint(target) + "\n");
}
```
OnHandlerEnter: 가상 명령어 디스패처가 다음 Opcode 핸들러로 분기 점프하는 런타임 순간에 가동됩니다. 인자로 목적지 주소(target)와 가상 Opcode 인덱스(opcode)를 받습니다. 새로운 핸들러 본문으로 진입했으므로 디스패처 노이즈 플래그를 해제하고(g_inDispatch = false), 통계 카운트를 증가시킨 뒤 진입 로그를 기록합니다.

```cpp
VOID PIN_CLASS::OnLea(ADDRINT ea, ADDRINT esp, BOOL isStack)
{
    if (!g_curHandler || g_inDispatch) return;
    LOG("      LEA   -> " + Loc(ea, esp, isStack) + "\n"); 
}
```
OnLea: LEA 명령어가 가상 핸들러 내에서 주소 산술 연산을 수행할 때 감지되어 들어옵니다. 인자로 계산된 유효 주소 결과값인 ea를 받습니다. LEA는 실제 메모리 R/W가 없는 가상 주소 연산이므로 별도의 대입 로그 없이 결과 주소 문자열만 추적 로그에 기록합니다.

```cpp
VOID PIN_CLASS::OnMemRead(ADDRINT addr, UINT32 size, ADDRINT esp, BOOL isStack)
{
    if (!g_curHandler) return;
    g_handlerStats[g_curHandler].readCount++;
    if (g_inDispatch) return;   // 디스패처 복호화 노이즈 스킵

    ADDRINT val = 0;
    // addr 유효 주소에서 실제 값을 안전하게 복사
    PIN_SafeCopy(&val, (VOID*)addr, (size < sizeof(ADDRINT)) ? size : sizeof(ADDRINT));

    LOG("      READ  " + Loc(addr, esp, isStack) +
        " val:" + StringFromAddrint(val) + ValTag(val, esp, isStack) + "\n");
}
```
OnMemRead: 타겟 명령어가 메모리 영역을 참조하여 읽어갈 때 호출됩니다. 인자로 전달받은 실제 주소 addr와 크기 size를 사용하며, 원시 포인터 참조 시 발생할 수 있는 크래시를 방지하기 위해 핀 엔진 고유의 안전 복사 API인 PIN_SafeCopy를 사용해 툴 내부 변수 val로 데이터를 인출해 온 뒤 기록합니다.

```cpp
VOID PIN_CLASS::OnMemWriteBefore(ADDRINT addr, UINT32 size, ADDRINT esp)
{
    g_wAddr = addr; g_wSize = size; g_wEsp = esp;
}

VOID PIN_CLASS::OnMemWriteAfter(BOOL isStack)
{
    if (!g_curHandler) return;
    g_handlerStats[g_curHandler].writeCount++;

    ADDRINT val = 0;
    PIN_SafeCopy(&val, (VOID*)g_wAddr, (g_wSize < sizeof(ADDRINT)) ? g_wSize : sizeof(ADDRINT));

    if (val == kDispatchMagic1 || val == kDispatchMagic2)
        g_inDispatch = true;

    if (g_inDispatch) return;   // 디스패처 노이즈 스킵

    LOG("      WRITE " + Loc(g_wAddr, g_wEsp, isStack) +
        " <- " + StringFromAddrint(val) + ValTag(val, g_wEsp, isStack) + "\n");
}
```
OnMemWriteBefore & OnMemWriteAfter: 메모리 쓰기 연산을 추적하기 위한 2-Step 분리 루틴입니다. 값이실제로 변경되기 전(Before)에 대상 주소 속성을 전역 변수 g_wAddr에 임시 저장하고, 대입 처리가 끝난 직후(After) 시점에 PIN_SafeCopy로 최종 수정본 데이터 val을 회수합니다. 이 과정에서 디스패처 시그니처 상수가 포착되면 g_inDispatch = true를 켜서 노이즈 수집을 즉시 중단합니다.

## [0x06] 계측 인터페이스 구현 (AnalyzeInstruction)
타겟 명령어 스트림을 가로채 어셈블리 단에 분석 지뢰를 매설하는 핵심 계측 루틴입니다.

```cpp
void PIN_CLASS::AnalyzeInstruction(const INS& ins)
{
    OPCODE op = INS_Opcode(ins);

    if (op == XED_ICLASS_PUSHAD)
        INS_InsertCall(ins, IPOINT_BEFORE, (AFUNPTR)TurnOnVmSwitch, IARG_END);
    else if (op == XED_ICLASS_POPAD)
    {
        INS_InsertIfCall(ins, IPOINT_BEFORE, (AFUNPTR)CheckIfVmActive, IARG_END);
        INS_InsertThenCall(ins, IPOINT_BEFORE, (AFUNPTR)TurnOffVmSwitch, IARG_END);
    }
```
PUSHAD/POPAD 인터페이스: XED_ICLASS_PUSHAD 명령어를 만나면 실행 직전(IPOINT_BEFORE)에 TurnOnVmSwitch 분석 함수를 주입하도록 INS_InsertCall로 명세합니다. POPAD를 계측할 때는 JIT 조건 붕괴 방지를 위해 핀 규칙대로 INS_InsertIfCall로 내부 상태 검사 함수(CheckIfVmActive)를 선행 조회시킨 후 참일 때만 TurnOffVmSwitch가 기동되도록 연동 설계했습니다.

```cpp
    if (CheckIfVmActive())
    {
        // 1. Dispatcher jmp [edi+eax*4] - Handler entry
        if (INS_IsIndirectControlFlow(ins) && !INS_IsRet(ins) &&
            INS_MemoryBaseReg(ins) == REG_EDI &&
            INS_MemoryIndexReg(ins) == REG_EAX &&
            INS_MemoryScale(ins) == 4)
        {
            INS_InsertCall(ins, IPOINT_TAKEN_BRANCH, (AFUNPTR)OnHandlerEnter,
                IARG_REG_VALUE, REG_EAX,

```
디스패처 구문 정밀 필터: 가상 머신의 핵심 분기문인 jmp [edi+eax*4]를 JIT 단에서 정확히 솎아내기 위한 핵심 가상화 필터 조건입니다. 간접 제어 흐름(INS_IsIndirectControlFlow)이고 리턴문이 아니며, 베이스 레지스터가 REG_EDI이고 인덱스 레지스터가 REG_EAX이며 스케일 팩터가 4 바이트(DWORD 크기)인지 검증합니다. 조건을 모두 충족하면 분기 점프가 완전히 일어난 직후인 IPOINT_TAKEN_BRANCH 시점에 콜백을 매설하여 목적지 타겟 주소(IARG_BRANCH_TARGET_ADDR)와 가상 Opcode 데이터(REG_EAX)를 인출해 냅니다.

```cpp
        // 2-a. lea: Check base register to profile stack pointer calculations accurately
        if (INS_IsLea(ins))
        {
            REG baseReg = INS_MemoryBaseReg(ins);
            BOOL isStackLea = (baseReg == REG_ESP || baseReg == REG_EBP);

            INS_InsertPredicatedCall(ins, IPOINT_BEFORE, (AFUNPTR)OnLea,
                IARG_EXPLICIT_MEMORY_EA,   ///< Address calculated by lea
                IARG_REG_VALUE, REG_ESP, IARG_BOOL, isStackLea, IARG_END);
        }

        // 2. Memory read (Source)
        if (INS_IsMemoryRead(ins))
        {
            BOOL isStackRead = INS_IsStackRead(ins); ///< Intel Pin API for stack read detection
            INS_InsertPredicatedCall(ins, IPOINT_BEFORE, (AFUNPTR)OnMemRead,
                IARG_MEMORYREAD_EA, IARG_MEMORYREAD_SIZE,
                IARG_REG_VALUE, REG_ESP, IARG_BOOL, isStackRead, IARG_END);
        }

        // 3. Memory write (Destination, value handled in AFTER)
        if (INS_IsMemoryWrite(ins))
        {
            BOOL isStackWrite = INS_IsStackWrite(ins); ///< Intel Pin API for stack write detection (Fixed Bug)
            INS_InsertPredicatedCall(ins, IPOINT_BEFORE, (AFUNPTR)OnMemWriteBefore,
                IARG_MEMORYWRITE_EA, IARG_MEMORYWRITE_SIZE,
                IARG_REG_VALUE, REG_ESP, IARG_END);

            if (INS_IsValidForIpointAfter(ins))
            {
                INS_InsertPredicatedCall(ins, IPOINT_AFTER, (AFUNPTR)OnMemWriteAfter, IARG_BOOL, isStackWrite, IARG_END);
            }
        }
    }
}
```
정밀 분석 주입 및 파라미터 규격: INS_InsertPredicatedCall 함수를 사용하여 조건부 명령어 실행 케이스에서도 안전하게 동작하도록 분석 콜백을 삽입합니다. 파라미터는 읽을 메모리 유효 주소(IARG_MEMORYREAD_EA), 연산 크기(SIZE), 연산 시점의 물리적 주소 공간 데이터인 스택 포인터 레지스터(IARG_REG_VALUE, REG_ESP)와 선행 판별해 둔 스택 전용 조작 플래그들(isStackRead, isStackWrite)을 명시적으로 전달합니다. 특히 쓰기 연산 시에는 핀 공식 API인 INS_IsStackWrite를 적용하여 정밀도를 대폭 끌어올렸습니다.

### 실전 덤프 로그 라인별 매칭 분석
위의 정교한 톱니바퀴들이 유기적으로 맞물려, 2부에서 추적했던 가상 레지스터 복구 첫 구간이 실제 덤프 로그 상에 어떻게 투영되는지 1:1로 매칭해 보겠습니다.

```text
 [tid:13904] [0x00000000] op(DECODING)=0x0000002d handler=0x0041f566
 [tid:13904]       READ  [Read OPCODE]0x004219b1 val:0x000000ed
```
로그 매칭 분석: AnalyzeInstruction이 포착한 디스패처 간접 분기문 점프 조건이 런타임에 실행되어 OnHandlerEnter 콜백이 가동된 결과 라인입니다. 0x0041f566 번지 가상 핸들러 본문으로 진입했음을 타겟 플래그로 명시합니다.

이어 가상 핸들러 코드 영역 내부에서 가상 명령 디코딩 연산을 수행하기 위해 난독화 바이트 데이터를 로드하는 행위가 OnMemRead 루틴에 의해 잡혀 READ [Read OPCODE] val:0x000000ed 형태로 정확히 검출됩니다.

```text
 [tid:13904]       LEA   -> [VM_EFLAGS]0x0041e61c
 [tid:13904]       WRITE [STACK]0x0019fde0 <- 0x0041e61c (->VM_EFLAGS)
```
로그 매칭 분석: 핸들러 코드 중간의 lea eax, [edi+0x1c] 명령어를 INS_IsLea 구문이 낚아채어 OnLea 루틴으로 매개변수를 유도한 지점입니다. 레지스터 내부 오프셋 연산을 수행한 RegionName 분석기가 해당 연산 결과 타겟 물리 주소가 VCPU 컨텍스트의 [VM_EFLAGS] 슬롯 메모리 번지임을 정확하게 맵핑해 냈습니다.

이어 이 연산된 구조체 내부 슬롯 주소 데이터를 다음 후행 가상 핸들러에게 전달하기 위해 가상 스택에 집어넣는 동작이 OnMemWriteAfter 2-Step 엔진에 의해 WRITE [STACK] <- 0x0041e61c (->VM_EFLAGS) 로그 파일 형태로 선명하게 매핑 출력됩니다.

```text
 [tid:13904] [0x00000002] op(DECODING)=0x00000032 handler=0x004202e0
 [tid:13904]       READ  [STACK]0x0019fde4 val:0x00000216 (->STACK)
 [tid:13904]       WRITE [VM_EFLAGS]0x0041e61c <- 0x00000216
```
로그 매칭 분석: 연산의 최종 마무리를 담당하는 0x004202e0 가상 핸들러 연산 지점입니다. 가상 머신 제어권 이관 최초 단계에 리얼 CPU가 백업하여 스택에 임시 보관해 두었던 원래의 정식 플래그 원본 값인 0x216 데이터를 스택 영역에서 인출하여 읽어 들입니다(READ [STACK] ... val:0x00000216).

최종 단계로, OnMemWriteAfter 콜백이 보관 주소를 해독하여 스택에서 복구해 낸 진짜 플래그 데이터 원본을 VCPU 구조체 메모리 내부의 [VM_EFLAGS] 컨텍스트 슬롯 내부 주소로 정확하게 대입 연산 완료(WRITE [VM_EFLAGS]0x0041e61c <- 0x00000216) 시키는 일련의 실행 흐름이 실증 라인 단위로 완벽하게 복원 및 증명되는 것을 명확히 확인할 수 있습니다.

## [0x07] Conclusion
결과적으로 단 한 줄의 어셈블리를 복원하기 위해 72번이나 뺑뺑이를 도는 가상화 루틴을, Intel Pin API를 통해 노이즈 없이 깔끔하게 덤프해 내는 자동화 툴을 완성했습니다.

단순한 툴 제작을 넘어, 인텔 핀의 JIT 컴파일 특성과 포인터 생명주기 제어, 가상 레지스터가 동기화되는 타이밍 제어, 그리고 로우레벨에서의 정밀한 스택/메모리 판별 기법까지 깊게 파고들 수 있었던 유익한 과정이었습니다.

이제 이 PinVMTracer가 뱉어내는 깨끗한 핸들러 통계와 R/W 로그를 무기로 삼아, 단순한 mov 대입 연산이 아닌 분기문과 함수 호출이 섞인 고난도 가상화 코드의 역연산(Devirtualization)을 다음 포스팅에서 본격적으로 다루어 보겠습니다.
