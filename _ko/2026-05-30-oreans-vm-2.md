## [0x00] Overview
 
[1부](https://leegwan.github.io/ko/posts/2026-05-25-oreans-vm-1/)에서는 가상화 바이너리가 `.VCPU` / `.VCODE`로 나뉘고, 제어 흐름이 Control Code → Main Handler → Opcode Handler로 돈다는 큰 그림까지 봤습니다. 그리고 마지막에 핸들러 트레이스 로그와 `End Result = 0x12345678`만 띄워놓고, "각 핸들러가 뭘 하는지는 다음에"라고 미뤄뒀습니다.
 
이번 글이 그 다음입니다. 트레이스에 찍힌 핸들러를 하나씩 디버거로 따라가며, 가상화된 코드가 핸들러 안에서 실제로 어떻게 굴러가는지 봤습니다. 솔직히 처음 트레이스만 봤을 때는 72줄이나 되는 핸들러 호출이 대체 뭘 하는 건지 감이 안 왔는데, 끝까지 따라가보니 결론은 허무할 정도로 단순했습니다. 이 72스텝(등장하는 고유 핸들러는 14종)이 결국 원본의 딱 한 줄이었습니다.
 
```asm
mov [ebp-8], 0x12345678
; 이후 0x411919로 복귀하여 [ebp-8] 값을 함수 인자로 사용
```
 
사람 눈엔 한 줄인데 VM 안에서는 핸들러 14종이 72번 불려가며 이 한 줄을 만들어냅니다. 1부에서 말한 "단순 연산 하나가 수십 단계로 팽창한다"가 핸들러 레벨에서 정확히 어떻게 일어나는지, 이번에 직접 확인했습니다.
 
 **Note:** 분석에 사용한 값(주소, 레지스터 등)은 디버깅 세션마다 달라집니다. 핵심은 절대 주소가 아니라 핸들러가 무엇을 하느냐이므로, 캡처값은 흐름을 따라가기 위한 참고로만 봐주시면 됩니다.
 
## [0x01] 핸들러를 읽는 두 가지 전제
 
핸들러를 까기 전에 두 가지만 먼저 정리합니다. 이걸 알고 보면 14종이 훨씬 빨리 눈에 들어옵니다.
 
**[-] 스택은 핸들러 사이의 통로다**
 
1부에서 봤듯 이 VCPU는 레지스터 머신이고, 실제 상태는 VM_Context 슬롯에 들어 있습니다. 그런데 핸들러가 그 슬롯에 직접 접근할 때 항상 스택을 한 번 거칩니다. 한 핸들러가 끝나며 `[esp]`에 주소나 값을 남기면, 다음 핸들러가 그걸 받아 자기 일을 합니다. 그래서 핸들러 두세 개가 모여야 의미 있는 동작 하나가 완성되는 경우가 대부분입니다.
 
**[-] 거의 모든 핸들러에 정크가 섞여 있다**
 
핸들러를 열어보면 `sub esp,4` 했다가 다시 되돌리고, `ebp`·`ebx`를 잠깐 빌렸다 복원하고, `not`·`and`·`xor`·`xchg` 연산을 넣었다가 결과를 버리는 코드가 잔뜩 들어 있습니다. 전부 분석을 방해하려는 난독화입니다. 아래에서는 이 정크를 걷어낸 **실제 동작**만 적습니다.
 
## [0x02] VM_Context 구조체
 
핸들러들이 공통으로 참조하는 VM_Context부터 확정하고 갑니다. 베이스는 `0x0041E600`이고 런타임엔 EDI가 이 주소를 가리킵니다. 1부에서는 레지스터 슬롯(+0x00 ~ +0x1C)까지만 정리했는데, 핸들러를 따라가다 보니 그 뒤쪽과 핸들러 테이블 위치까지 채울 수 있었습니다. 실제 덤프는 이렇습니다.
 
```
0041E600  E4 FE 19 00  E4 FE 19 00  00 30 38 00  CC CC CC CC
0041E610  48 13 41 00  88 7B 69 00  00 00 00 00  16 02 00 00
0041E620  00 00 00 00  00 00 00 00  06 00 00 00  00 00 00 00
0041E630  01 00 00 00  00 00 00 00  0E 0D 42 00  88 F6 41 00
```
 
DWORD 단위로 해석하면 다음과 같습니다.
 
| 오프셋 | 필드 | 캡처 값 | 비고 |
|--------|------|---------|------|
| +0x00 | VM_EBP | 0019FEE4 | |
| +0x04 | VM_EDI | 0019FEE4 | |
| +0x08 | VM_EBX | 00383000 | |
| +0x0C | VM_EAX | CCCCCCCC | |
| +0x10 | VM_ESI | 00411348 | |
| +0x14 | VM_EDX | 00697B88 | |
| +0x18 | VM_ECX | 00000000 | |
| +0x1C | VM_EFLAGS | 00000216 | 트리오가 가장 먼저 채우는 슬롯 |
| +0x20 | ? | 0 | 용도 확인 안 됨 |
| +0x24 | ? | 0 | 용도 확인 안 됨 |
| +0x28 | ? | 6 | `[15]` 핸들러가 쓰는 값, 이후 읽는 곳을 못 찾음 |
| +0x2C | ? | 0 | `[42]` 핸들러가 리턴값에 더하는 값 |
| +0x30 | init_flag | 1 | Control Code의 `[edi+30]` lock cmpxchg 대상 |
| +0x34 | ? | 0 | 핸들러 테이블 직전 경계 |
| +0x38~ | handler_table[] | 0x420D0E, 0x41F688, … | Opcode 핸들러 주소 배열 |
 
```c
// 베이스 = 0x0041E600 (런타임엔 EDI가 가리킴)
struct VM_CONTEXT {
    uint32_t VM_EBP;        // +0x00
    uint32_t VM_EDI;        // +0x04
    uint32_t VM_EBX;        // +0x08
    uint32_t VM_EAX;        // +0x0C
    uint32_t VM_ESI;        // +0x10
    uint32_t VM_EDX;        // +0x14
    uint32_t VM_ECX;        // +0x18
    uint32_t VM_EFLAGS;     // +0x1C
 
    uint32_t unk_20;        // +0x20  미상
    uint32_t unk_24;        // +0x24  미상
    uint32_t unk_28;        // +0x28  [15]가 6을 씀, 이후 액세스 없음
    uint32_t unk_2C;        // +0x2C  [42]가 리턴값에 가산 (이번 캡처값 0)
    uint32_t init_flag;     // +0x30  Control Code lock cmpxchg 대상
    uint32_t unk_34;        // +0x34  미상
 
    void*    handler_table[]; // +0x38~  Main Handler가 jmp [edi+eax*4]로 분기
};
```
 
핸들러 테이블은 `+0x38`부터입니다. 덤프를 보면 `+0x38 = 0x00420D0E`, `+0x3C = 0x0041F688`처럼 주소 값들이 들어 있고, Main Handler가 `jmp [edi+eax*4]`로 이 테이블을 인덱싱해 핸들러로 분기합니다.
 
## [0x03] 핸들러 카탈로그
 
트레이스 72스텝이 전부 아래 14종의 조합입니다. 정크를 걷어낸 실제 동작만 적습니다.
 
| 핸들러 | 인덱스(EAX) | 실제 동작 |
|--------|-------------|----------|
| `0x41f566` | 0x2d | `.VCODE` 1바이트를 읽어 디코딩, `lea eax,[edi+eax*4]`로 VM 레지스터 슬롯 주소를 구해 스택에 올림 |
| `0x421482` | 0x89 | `mov edx,[esp]` — 스택의 주소/값을 edx로 |
| `0x4202e0` | 0x32 | `pop dword ptr ds:[edx]` — 스택의 값을 타깃 주소에 저장 |
| `0x4202ce` | 0x58 | VM_Context `+0x28`에 6을 씀 (이후 액세스 없음) |
| `0x41ec7e` | 0x76 | `mov edx, esp` — 현재 스택 주소를 캡처 |
| `0x41fae3` | 0x9b | `mov [esp], edx` — 주소/값을 스택에 적재 |
| `0x41fb68` | 0x5e | `lodsd`로 `.VCODE` 4바이트를 읽어 `xor`+`add` 누적으로 복호화 후 `mov [esp],eax` |
| `0x41fdb4` | 0x1d | `mov eax,[esp]` + esi 경유 보정 + `add [esp],eax` — base + offset 주소 계산 |
| `0x4218bf` | 0x75 | `mov esp,[esp]` — 작업 esp를 계산된 주소로 전환 |
| `0x4207e6` | 0x33 | `push dword ptr ds:[edi+eax*4]` — VM_Context 슬롯값을 스택에 올림 |
| `0x4208b2` | 0x94 | 리턴값 `+= [VM_Context+0x2C]` |
| `0x41ee2a` | 0x7a | `lodsd` + `add esi,eax`로 ESI에 읽은 값을 더함 |
| `0x41fb23` | 0x25 | `push dword ptr ds:[edx]` — 슬롯이 가리키는 값을 스택에 push |
| `0x41f6a6` | 0x1a | 스택 정리 + `popad`/`popfd` + `ret` — Real CPU 복구 후 복귀 |
 
트레이스를 핸들러 패턴으로 끊으면 다섯 구간으로 나뉩니다. 0x04부터 차례대로 보겠습니다.
 
| 구간 | 트레이스 | 하는 일 |
|------|----------|---------|
| Phase 1 | `[0]~[27]` | 백업 레지스터를 VM_Context 슬롯에 저장 |
| Phase 2 | `[28]~[40]` | `[ebp-8]` 주소를 만들고 복호화된 `0x12345678`을 기록 |
| Phase 3 | `[41]~[43]` | 리턴 어드레스 복호화·세팅, `add esi,eax`로 ESI 변경 |
| Phase 4 | `[44]~[70]` | VM_Context 슬롯값을 스택으로 복원 |
| Phase 5 | `[71]` | `popad`/`popfd` + `ret`로 복귀 |
 
## [0x04] Phase 1 — 레지스터를 VM_Context에 채우기
 
1부 트레이스에서 제일 먼저 눈에 띄던 `0x41f566 → 0x421482 → 0x4202e0` 트리오 구간입니다. 한 사이클이 곧 "VM_Context 슬롯에 값 하나 저장"입니다.
 
### [-] 첫 트리오 따라가기
 
진입 직후 스택 위에는 백업된 레지스터 값(첫 사이클은 EFLAGS=0x216)이 올라가 있고, 트리오가 그걸 슬롯으로 옮깁니다.
 
![0x41f566](/assets/img/Code Virtualization/Code Virtualization-2/0x41f566.png)
 
`0x41f566`은 `lodsb`로 1바이트를 읽고 `sub al,bl`로 디코딩한 다음, `movzx eax,al`로 인덱스를 만들고 `lea eax,[edi+eax*4]`로 슬롯 주소를 구해 `mov [esp],eax`로 스택에 올립니다. 첫 사이클 결과는 EAX = 0x0041E61C(베이스 + 0x1C, 즉 EFLAGS 슬롯)이고, 스택에 `0019FE04 = 0041E61C`가 남습니다.
 
![0x421482](/assets/img/Code Virtualization/Code Virtualization-2/0x421482.png)
 
`0x421482`는 `mov edx,[esp]`로 그 슬롯 주소를 edx에 담습니다(EDX = 0041E61C).
 
![0x4202e0](/assets/img/Code Virtualization/Code Virtualization-2/0x4202e0.png)
 
`0x4202e0`은 `pop dword ptr ds:[edx]`로 스택의 값을 슬롯에 씁니다. 디버거 하단에 `[edx]=[0041E61C]=216`이 찍히고, 덤프의 `0041E61C`가 `16 02 00 00`(=0x216)으로 채워집니다.
 
세 핸들러를 합치면 이 한 줄입니다.
 
```asm
mov [VM_Context+0x1C], 0x216   ; EFLAGS 백업값을 슬롯에 저장
```
 
`0x41f566`이 **어디에**(슬롯 주소), `0x421482`가 그 주소를 작업 레지스터로 확정, `0x4202e0`이 비로소 **무엇을**(스택 값) 그 주소에 씁니다. 1부 OPCODE Handlers에서 본 "주소 준비 → 주소 확정 → 값 기록" 3박자 그대로입니다.
 
### [-] 9번의 반복
 
이 트리오는 `[0]~[27]`에서 9번 반복되며, Control Code가 백업해둔 레지스터 값들을 슬롯에 하나씩 채웁니다. 중간 `[15]`의 `0x4202ce`만 트리오에서 벗어나는데, VM_Context `+0x28`에 6을 쓰지만 이후 어디에서도 그 값을 읽지 않습니다. 솔직히 왜 넣는지는 이 트레이스만으로는 모르겠어서 미상으로 둡니다.
 
## [0x05] Phase 2 — mov [ebp-8], 0x12345678
 
이 글에서 제일 중요한 구간입니다. 쓸 **값(0x12345678)** 과 쓸 **주소([ebp-8])** 를 둘 다 `.VCODE`에서 복호화해 만들어냅니다.
 
### [-] 값을 만들어 임시로 놓기 ([28]~[33])
 
![0x41ec7e](/assets/img/Code Virtualization/Code Virtualization-2/0x41ec7e.png)
 
`[28]` `0x41ec7e` — `sub esp,4`·`mov [esp],esp`·`add [esp],4`·`mov edx,[esp]`·`add esp,4`. 정크 코드를 걷어내면 실제로는 현재 스택 주소를 edx에 담습니다(`mov edx, esp`). 결과 EDX = 0019FE08.
 
![0x41fae3](/assets/img/Code Virtualization/Code Virtualization-2/0x41fae3.png)
 
`[29]` `0x41fae3` — `mov [esp], edx`로 그 주소를 다시 스택에 올립니다(`[0019FE04] = 0019FE08`).
 
![0x41fb68](/assets/img/Code Virtualization/Code Virtualization-2/0x41fb68.png)
 
`[30]` `0x41fb68` — `lodsd`로 `.VCODE` 4바이트를 읽어 `xor eax,ebx` 후 여러 `add`로 누적 복호화합니다. 이 사이클 결과는 0x00000004(`0019FE00 = 00000004`).
 
`[31]` `0x41fdb4` — `mov eax,[esp]`(=4)로 오프셋을 받고, esi를 경유해 보정한 뒤 `add [esp],eax`로 base에 오프셋을 더해 `esp+4` 위치를 계산합니다.
 
![0x4218bf](/assets/img/Code Virtualization/Code Virtualization-2/0x4218bf.png)
 
`[32]` `0x4218bf` — `mov esp, dword ptr ss:[esp]`로 작업 esp를 그 주소로 전환합니다.
 
![0x41fb68 값 복호화](/assets/img/Code Virtualization/Code Virtualization-2/0x41fb68_값복호화.png)
 
`[33]` `0x41fb68` — 같은 디코더가 다시 호출되며, 이번엔 암호화돼 있던 값이 복호화되어 0x12345678이 됩니다(`EAX = 12345678`). 다만 이 시점의 0019FE08은 최종 목적지가 아니라 중간 스크래치입니다.
 
### [-] 진짜 주소를 계산해서 저장 ([34]~[40])
 
![0x4207e6](/assets/img/Code Virtualization/Code Virtualization-2/0x4207e6.png)
 
`[34]` `0x4207e6` — `push dword ptr ds:[edi+eax*4]`로 VM_Context에 저장해둔 EBP 값을 스택에 올립니다. 레지스터를 직접 안 쓰고 컨텍스트에서 꺼내오는 게 1부에서 본 레지스터 머신 구조 그대로입니다.
 
`[35]` `0x421482` → `mov edx,[esp]`, `[36]` `0x41fae3` → `mov [esp],edx`로 값을 옮깁니다.
 
`[37]` `0x41fb68` — 디코더가 또 호출되어 `.VCODE`에서 읽은 값을 복호화하니 0xFFFFFFF8, 2의 보수로 **-8**입니다(`0019FE00 = FFFFFFF8`).
 
![0x41fdb4](/assets/img/Code Virtualization/Code Virtualization-2/0x41fdb4.png)
 
`[38]` `0x41fdb4` — `mov eax,[esp]`(=0xFFFFFFF8)와 EBP를 더합니다. `0019FEE4(EBP) + (-8) = 0019FEDC`, 곧 `[ebp-8]`의 실제 주소입니다.
 
`[39]` `0x421482` — 그 주소(0019FEDC)를 edx로 가져옵니다.
 
![0x4202e0](/assets/img/Code Virtualization/Code Virtualization-2/0x4202e0.png)
 
`[40]` `0x4202e0` — `pop dword ptr ds:[edx]`로 스택의 값을 그 주소에 씁니다. EDX = 0019FEDC, pop되는 값 = 0x12345678. 덤프의 `0019FEDC`가 `12345678`로 채워집니다.
 
정리하면 `[28]~[39]`가 EBP-8(= 0019FEDC) 주소를 만들고, `[40]`이 거기에 0x12345678을 씁니다.
 
```asm
mov [ebp-8], 0x12345678
```
 
여기서 짚고 갈 게 있습니다. 저장할 값(0x12345678)뿐 아니라 **주소 오프셋(-8)조차 평문으로 안 두고** `.VCODE`에 암호화해 뒀다가 런타임에 `0x41fb68`로 풀어서 `[ebp-8]`을 만든다는 겁니다. 상수도 오프셋도 디코더 핸들러를 거쳐야만 드러납니다. 이게 정적 분석을 짜증나게 만드는 포인트입니다.
 
## [0x06] Phase 3 — 리턴 준비
 
![0x41fb68 리턴 어드레스](/assets/img/Code Virtualization/Code Virtualization-2/0x41fb68.png)
 
`[41]` `0x41fb68` — 디코더가 이번엔 리턴 어드레스 0x411919를 복호화해 스택에 세팅합니다(`0019FE08 = 00411919`). 코드 영역을 같이 보면 0x411919에 `mov eax, dword ptr ss:[ebp-8]`이 있고 디버거가 "ret address" 주석까지 달아둡니다. 즉 가상화 블록을 빠져나오면 이 주소로 복귀해서, 방금 써둔 `[ebp-8] = 0x12345678`을 다시 읽게 됩니다.
 
![0x4208b2](/assets/img/Code Virtualization/Code Virtualization-2/0x4208b2.png)
 
`[42]` `0x4208b2` — `edx = edi+0x2C`로 VM_Context `+0x2C` 주소를 잡고, 그 값을 읽어 리턴 어드레스에 더합니다(디버거 주석 `read vm_context+2c`). 실제 동작은 `[리턴 어드레스] += [VM_Context+0x2C]`입니다. 다만 이번 캡처에서는 `+0x2C = 0`이라 0x411919가 그대로입니다.
 
 **Note:** `+0x2C`가 0이 아닌 다른 경로에서 어떤 보정을 하려는 건지는 이 트레이스 하나만으로는 단정할 수 없습니다. 일단 "이번 케이스에선 가산값이 0이라 불변"까지만 기록합니다. 다만 개인적으로는, ASLR로 이미지 베이스가 바뀌었을 때 그 델타를 리턴 어드레스에 더해 보정하기 위한 자리가 아닐까 추측합니다(어디까지나 추측입니다).
 
![0x41ee2a](/assets/img/Code Virtualization/Code Virtualization-2/0x41ee2a.png)
 
`[43]` `0x41ee2a` — `lodsd`로 `.VCODE`에서 값을 읽고(`01 00 00 00` → 1) `add esi,eax`로 ESI에 그 값을 더합니다. 같이 보이는 `mov ebx`·`not`·`and`·`xor`·`push`/`pop edi`는 결과를 안 쓰는 정크입니다.
 
## [0x07] Phase 4 — VM_Context를 스택으로 되돌리기
 
이 구간 트리오는 `0x41f566 → 0x421482 → 0x41fb23`입니다. 앞 둘은 Phase 1과 같고 **마지막만** `0x4202e0`(저장)에서 `0x41fb23`(읽어서 push)으로 바뀝니다.
 
![0x41fb23](/assets/img/Code Virtualization/Code Virtualization-2/0x41fb23.png)
 
첫 사이클로 보면, `[44]` `0x41f566`이 EFLAGS 슬롯 주소(0x0041E61C)를 스택에 올리고, `[45]` `0x421482`가 그걸 edx에 담고(EDX = 0041E61C), `[46]` `0x41fb23`이 `push dword ptr ds:[edx]`로 슬롯값을 스택에 push합니다. 하단에 `[edx]=[0041E61C]=216`이 찍히며 EFLAGS 값이 스택에 쌓입니다.
 
Phase 1과 정확히 반대 방향입니다. Phase 1이 "스택 → 슬롯(저장)"이었다면 여기는 "슬롯 → 스택(복원)"입니다. 이 트리오가 `[44]~[70]`에서 9번 반복됩니다.
 
![거의 막바지 스택 상태](/assets/img/Code Virtualization/Code Virtualization-2/거의막바지.png)
 
스택을 보면 위쪽에 `[41]`에서 세팅한 리턴 어드레스(0x411919)가 있고, 그 아래로 `[44]~[70]`이 복원한 레지스터 값들(EFLAGS=0x216, EBP=0019FEE4 등)이 차곡차곡 쌓입니다. 결국 VM_Context에 백업해둔 레지스터를, 가상화 블록을 빠져나가기 전에 다시 스택에 재구성하는 작업이고, 이건 곧 이어질 `popad`/`popfd`를 위한 준비입니다.
 
## [0x08] Phase 5 — popad/popfd로 빠져나오기
 
![0x41f6a6](/assets/img/Code Virtualization/Code Virtualization-2/0x41f6a6.png)
 
`[71]` `0x41f6a6` — 스택을 정리한 뒤 이걸 합니다.
 
```asm
sub dword ptr ds:[edi+30], 6AE5   ; VM_Context +0x30 슬롯
popad                             ; [44]~[70]이 쌓은 범용 레지스터 8개 복구
popfd                             ; EFLAGS 복구
ret                               ; 스택 top의 리턴 어드레스(0x411919)로 복귀
```
 
`popad`/`popfd`로 Real CPU 레지스터와 플래그를 한 번에 복구하고, `ret`으로 0x411919로 복귀합니다. 1부 Control Code의 `pushad`/`pushfd` 진입과 대칭되는 출구입니다. 앞에 `sub dword ptr ds:[edi+30], 6AE5`로 `+0x30` 슬롯도 한 번 건드립니다.
 
복귀 직후 EIP는 0x411919를 가리키고, 그 자리의 `mov eax, dword ptr ss:[ebp-8]`이 방금 써둔 `[ebp-8] = 0x12345678`을 읽습니다. 이어서 `push 417B30`("%d") → `call 411046` 흐름으로, 그 값을 함수 인자로 넘깁니다.
 
## [0x09] 전체 흐름 정리
 
| 구간 | 트레이스 | 요약 |
|------|----------|------|
| Phase 1 | `[0]~[27]` | 백업 레지스터를 VM_Context 슬롯에 저장 (트리오 `0x41f566 → 0x421482 → 0x4202e0` ×9, `[15]`는 `+0x28`에 6) |
| Phase 2 | `[28]~[40]` | 값(0x12345678)과 오프셋(-8)을 복호화, `[ebp-8]`에 저장 → `mov [ebp-8], 0x12345678` |
| Phase 3 | `[41]~[43]` | 리턴 어드레스 0x411919 복호화·세팅, `+[VM_Context+0x2C]`(=0)로 불변, `add esi,eax`로 ESI 변경 |
| Phase 4 | `[44]~[70]` | VM_Context 슬롯값을 스택으로 복원 (트리오 `0x41f566 → 0x421482 → 0x41fb23` ×9) |
| Phase 5 | `[71]` | `popad`/`popfd` + `ret` → 0x411919로 복귀 |
 
원본으로 돌아가면 결국 이 한 줄입니다.
 
```asm
mov [ebp-8], 0x12345678
; 0x411919로 복귀 → [ebp-8]을 읽어 push → call (함수 인자로 사용)
```
 
## [0x0A] Conclusion
 
상수 하나를 지역 변수에 넣고 함수에 넘기는 동작이, 핸들러 14종을 72번 호출하는 흐름으로 펼쳐졌습니다. 값도 오프셋도 리턴 어드레스도 전부 `.VCODE`에 암호화해 뒀다가 런타임에 디코더 핸들러로 풀어 쓰기 때문에, 한 줄을 복원하는 데도 핸들러를 끝까지 따라가야 합니다. 1부에서 말한 "분석 시간을 기하급수적으로 늘린다"는 목적이 핸들러 레벨에서 이렇게 실현되는 셈입니다.
 
다만 `+0x20`·`+0x24`·`+0x28`(6)·`+0x2C`·`+0x34` 슬롯의 정확한 용도, `[42]`가 더하는 `+0x2C`가 0이 아닐 때의 의미처럼 트레이스 하나로는 단정 못 하는 부분은 미해결로 남겨뒀습니다. 다음엔 분기·반복·함수 호출이 섞인 더 복잡한 코드 조각을 분석하면서 이 빈칸들을 채워보겠습니다.
